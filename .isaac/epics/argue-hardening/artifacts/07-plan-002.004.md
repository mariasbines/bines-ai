### CREATE `src/lib/argue-filter/haiku.ts`

Exports `classifyConversation(messages)` with signature:

```typescript
export async function classifyConversation(
  messages: Array<{ role: 'user' | 'assistant'; content: string }>,
): Promise<ArgueVerdict>
```

Flow:
1. Build a short classifier system prompt. Shape:
   ```
   You classify conversations for a personal website's chat. Return JSON only. Never include prose or markdown code fences.

   JSON shape:
   {
     "harm": "none" | "hate" | "threat" | "sexual" | "violence" | "self_harm",
     "off_brand": Array<"electoral_politics" | "hot_button_social" | "race_identity_politics" | "religion" | "named_private_people" | "family_beyond_site" | "conspiracy_crypto_hype">,
     "reasoning": string (optional, <= 500 chars)
   }

   Err toward allow on borderline turns. Only flag off_brand when the topic is clearly in one of the seven categories.

   Be strict on harm — real threats, hate speech, sexual content, violence, self-harm go in the harm field.
   ```
   Errs toward allow on borderline (ARGUE-BRD-002 mitigation).
2. Serialise the conversation as a single user message (role: 'user'). The SDK expects at least
   one user message.
3. Start an `AbortController` with a 3-second timeout.
4. Call `client.messages.create({ model: FILTER_MODEL, max_tokens: FILTER_MAX_TOKENS, system, messages, signal })`.
   (The SDK respects AbortSignals.)
5. Extract the response text from `response.content[0].text` (the Anthropic content-block shape).
6. `JSON.parse` the text.
7. Validate with `ARGUE_VERDICT.safeParse(parsed)`.
8. On success: return `parsed.data`.
9. On ANY failure (timeout, parse, schema, network): log
   `[argue-filter] classifier-error: <reason>` via `console.error` and return the fail-open
   verdict `{ harm: 'none', off_brand: [], reasoning: 'classifier_error' }`.

Why fail-open: ARGUE-SEC-002 mitigation — Haiku outage must not block the chat. The tightened
Sonnet prompt (002.003) is the belt; Haiku is the braces.

Module-level guard: the SDK client comes from `getAnthropicClient()`. Reuse that accessor so
`ANTHROPIC_API_KEY` handling stays in one place. If `getAnthropicClient()` throws (missing key),
treat it as a classifier error → fail-open (but the route also reads the key; the route will
surface the 500 before we get here in practice).

11. **NEW**: Otherwise (on-brand): start the Sonnet stream as before.
    - Wrap the stream in a `TransformStream` that:
      - On each chunk: pass through AND append assistant text to an accumulator. Parse the bytes
        as UTF-8, extract `text_delta` fields from `content_block_delta` events, accumulate.
      - On flush/close: build the full `ArgueLogEntry` with the accumulated assistant reply as a
        second turn, and schedule `appendArgueLog(entry)` via `after()`.
    - Return the piped response with the SAME headers the existing path uses.

Tee-through design: the upstream Anthropic stream emits SSE frames. The existing client parser
reads `data: ` lines from this stream. We need to pass them through verbatim AND parse them
ourselves to accumulate the assistant text. Use a `TransformStream<Uint8Array, Uint8Array>`:

```typescript
const accumulator: string[] = [];
const decoder = new TextDecoder();
let buffer = '';

const transform = new TransformStream<Uint8Array, Uint8Array>({
  transform(chunk, controller) {
    controller.enqueue(chunk); // pass through unchanged
    buffer += decoder.decode(chunk, { stream: true });
    let nl: number;
    while ((nl = buffer.indexOf('\n')) >= 0) {
      const line = buffer.slice(0, nl).trim();
      buffer = buffer.slice(nl + 1);
      if (!line.startsWith('data: ')) continue;
      try {
        const evt = JSON.parse(line.slice(6));
        if (evt?.type === 'content_block_delta' && evt?.delta?.type === 'text_delta') {
          accumulator.push(evt.delta.text ?? '');
        }
      } catch { /* skip */ }
    }
  },
  flush() {
    const assistantContent = accumulator.join('');
    const entry: ArgueLogEntry = { /* … */, turns: [...messages, { role: 'assistant', content: assistantContent }] };
    after(async () => {
      try { await appendArgueLog(entry); } catch (err) {
        console.error('[chat] log-append failed:', err instanceof Error ? err.name : 'unknown');
      }
    });
  },
});

return new Response(stream.toReadableStream().pipeThrough(transform), {
  headers: { 'Content-Type': 'text/event-stream', ...GOVERNED_BY_HEADER },
});
```

Client-abort semantics: if the client aborts mid-stream, the underlying `ReadableStream` is
cancelled, which closes the `TransformStream` writable side, which fires `flush()`. The partial
assistant content is logged. This matches AC-006 (no `truncated` flag, partial is fine).

**L-1 caveat**: on certain Node/Edge runtime drops, `flush` may not fire on cancel (spec
permits). Confirmed behaviour in the Next.js Edge runtime per Anthropic's streaming docs — the
transform's `flush()` fires on normal close; on `cancel` we route via a separate `cancel()` hook.
I'll wire both `flush` and `cancel` to trigger the log-append (idempotent via a single-fire
latch).

## Risks to watch

- **Timer / Abort interplay**: `AbortController.abort()` on the classifier must propagate through
  the SDK. The SDK accepts an AbortSignal in its request options. Timeout implementation: use
  `setTimeout` + `controller.abort('timeout')`. Test mocks reject with `{ name: 'AbortError' }`.
- **`after()` mocking**: `next/server` is imported by Next.js internals, so the mock needs to
  preserve everything else from the real module. Use `vi.mock('next/server', async (importOriginal) => { const mod = await importOriginal(); return { ...mod, after: vi.fn((cb) => afterCallbacks.push(cb)) }; })`.
- **TransformStream on Edge**: Web Streams API is available in Edge runtime (Next.js edge uses
  the V8/Miniflare-shaped runtime). Confirmed by existing `refusalEventStream()` in 002.003 using
  `ReadableStream`.
- **Partial content on abort**: `flush` may not fire on cancel. Mitigation: hook both `flush` and
  `cancel` (via the `cancel` method on the underlying transform's writable/readable). Single-fire
  latch prevents double-log.
- **Zero-length assistant turn on abort-before-any-text**: the log entry still gets written with
  an empty assistant content. Acceptable: reflects reality.
- **Salt env absence in tests**: all existing tests pass without `ARGUE_LOG_IP_SALT_CURRENT` in
  env. Adding the new 500-upstream branch means tests that DON'T set the salt will now return
  500 unless I set it in the beforeEach. Fix: set `process.env.ARGUE_LOG_IP_SALT_CURRENT` in the
  shared beforeEach; keep one dedicated test that deletes it to verify the 500-upstream branch.
- **classifier called once per request** — tests assert `mockMessagesCreate` called exactly once.
