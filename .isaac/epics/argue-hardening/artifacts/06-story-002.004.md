# Story 002.004: Haiku classifier + route integration

## Description

The second big story. Builds the Haiku 4.5 pre-flight classifier that emits a structured JSON verdict (harm + off_brand), then wires it into `/api/chat/route.ts` along with the post-stream log-append scheduling via `after()` from `next/server`. This is where 002.001 (storage) and 002.003 (refusal) meet to form the complete hardened chat path.

Fail-open on any classifier error — Anthropic outage must not DOS the chat. Sonnet's tightened system prompt (shipped in 002.003) is the belt that catches content when the classifier can't.

Sequential architecture (Haiku → Sonnet) is the v1 choice. Post-launch revisit trigger (parallel execution) fires only if observed p95 `latency_ms.pre_flight` exceeds 400ms — this is a watch-metric, not a v1 work item.

## Acceptance Criteria

- [ ] AC-001: `src/lib/chat/anthropic.ts` exports two new constants (existing exports unchanged): `FILTER_MODEL` (defaults to `claude-haiku-4-5`, overridable via `process.env.FILTER_MODEL`) and `FILTER_MAX_TOKENS = 256`.
- [ ] AC-002: `src/lib/argue-filter/haiku.ts` exports `classifyConversation(messages)` which: (a) calls the Anthropic SDK `messages.create` with `FILTER_MODEL`, (b) constructs a classifier system prompt that instructs the model to emit ONLY a JSON object matching the verdict shape, (c) includes the full conversation as the user turn, (d) enforces a 3-second timeout via `AbortController`.
- [ ] AC-003: `classifyConversation` parses the model's response as JSON, validates against the `ARGUE_VERDICT` zod schema (defined in `src/lib/argue-log/schema.ts` from 002.001), and returns the verdict. On any failure (timeout, parse error, schema rejection, SDK error), returns the fail-open verdict `{harm: 'none', off_brand: [], reasoning: 'classifier_error'}` and logs `[argue-filter] classifier-error: <reason>` via `console.error` (mitigates ARGUE-SEC-002).
- [ ] AC-004: `src/app/api/chat/route.ts` is modified to insert the new filter-and-log flow between agent-guard (step 5) and Sonnet stream (step 7 in the existing numbered comments). New flow:
  - Hash the IP using `hashIp()` from 002.001 + `ARGUE_LOG_IP_SALT_CURRENT`. If env var unset, return 500 with category `upstream` and log `[chat] salt-unconfigured` (mitigates ARGUE-SEC-003).
  - Call `classifyConversation()`; capture `latency_ms.pre_flight`.
  - If `verdict.off_brand.length > 0`: build `ArgueLogEntry` with `refused: true` and `model: ''`; schedule `appendArgueLog(entry)` via `after()` from `next/server`; return a response whose body is `refusalEventStream()` from 002.003, with `Content-Type: text/event-stream` and the existing `X-Governed-By` header.
  - Otherwise: proceed to Sonnet stream. Tee the stream through a `TransformStream` that copies chunks to the response body AND accumulates assistant tokens. On `flush`, build the `ArgueLogEntry` with the full conversation + verdict + captured latencies + `refused: false`, and schedule `appendArgueLog(entry)` via `after()`.
- [ ] AC-005: `after()` is imported from `next/server` and is the only mechanism scheduling log-append work. Log writes are never `await`-ed inside the request handler's main path — visitor latency is unaffected (mitigates ARGUE-PRF-001).
- [ ] AC-006: On client-abort mid-stream, the `TransformStream`'s `flush` (or the underlying `close`) still fires with whatever assistant tokens were emitted. The partial assistant content is logged as-is. No `truncated` flag in v1 (per open-item resolution in Phase 6).
- [ ] AC-007: Rate-limit step order is unchanged — rate-limit sits BEFORE the classifier call, so a bad actor cannot burn Haiku budget faster than Sonnet (confirms elicitation Q9).
- [ ] AC-008: The existing `X-Governed-By: bines.ai` header is on every response (refusal path AND stream path AND error paths), matching the existing chat route convention.
- [ ] AC-009: No Blob URL appears in any route response, `console.log`, `console.error`, or thrown error message (mitigates ARGUE-SEC-001).
- [ ] AC-010: `src/app/api/chat/__tests__/route.test.ts` is extended with new test cases covering: (a) off-brand verdict → refusal SSE + log scheduled with `refused: true`; (b) on-brand verdict → Sonnet stream flows + log scheduled on close with full assistant content; (c) classifier error → fail-open, Sonnet stream starts normally, log entry has `reasoning: 'classifier_error'`; (d) missing salt env var → 500 upstream; (e) `after()` is called exactly once per request on the happy path and the refusal path; (f) `x-forwarded-for` → `hashIp()` produces a stable 64-char hex that does not equal the raw IP. Existing tests continue to pass without modification.
- [ ] AC-011: `src/lib/argue-filter/__tests__/haiku.test.ts` covers: success path (valid JSON, valid schema), JSON-parse failure → fail-open, schema-invalid output → fail-open, AbortController timeout → fail-open, SDK throws → fail-open. All four fail-open paths log `[argue-filter] classifier-error`.

## Scope rules

- Do NOT add parallel Haiku/Sonnet execution. Sequential for v1 (revisit trigger documented, not a code path).
- Do NOT introduce streaming verdict surfacing to the client. Verdict is internal.
- Do NOT hard-block on classifier errors — fail-open is the documented policy.
- Do NOT log message content via `console.*` — it's already in the Blob log. Keep `console.error` messages to metadata only (category, latency, error reason).
- Do NOT add retry logic on Haiku errors — a single timeout + fail-open is correct; retries add latency without real benefit.
- Do NOT modify the client-side chat code (`src/components/ChatInterface.tsx`) — the refusal SSE intentionally matches the stream shape the client already parses.

## Risk mitigations (from 04-assessment.md)

- **ARGUE-SEC-002** Haiku fail-open → AC-003 (documented policy), AC-011 (test-verified)
- **ARGUE-SEC-003** Salt misconfig → AC-004 (fail-loud on missing current salt)
- **ARGUE-SEC-007** Prompt injection against classifier → AC-003 (zod validation on output rejects non-shape responses)
- **ARGUE-BRD-002** False-positive refusals → classifier prompt errs toward "allow" for borderline (noted in Technical Tasks §2)
- **ARGUE-BRD-003** False-negative → AC-004 (Sonnet stream still governed by the tightened system prompt from 002.003)
- **ARGUE-PRF-001** Latency creep → AC-005 (`after()` keeps log off user latency), 3-second Haiku timeout, 256 token cap
- **ARGUE-OPS-002** Haiku cost creep → rate-limit step order (AC-007) + existing spend alerts
- **ARGUE-OPS-004** No alerting on filter → AC-003 log signal for grep-based observability
