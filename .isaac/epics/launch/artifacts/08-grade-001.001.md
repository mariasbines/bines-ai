## Findings

### High — Must Fix Before Execute

#### H-001: Wrong monospace font (Alignment)

**Issue:** Plan installs and configures **JetBrains Mono** throughout (Step 3 `pnpm add -D`, Step 5 layout, Step 6 `globals.css` `--font-mono` mapping). The architecture doc specifies **IBM Plex Mono** in three places (lines 297, 303, 547 of `03-architecture.md`), including in the canonical dependency table and the font pairing notes.

**Impact:** Shipping JetBrains Mono deviates from the locked typography decision. The Fieldwork data-strip aesthetic ("README/dashboard" look) was chosen with IBM Plex Mono's character set in mind. Swapping it after content is authored is a more disruptive change than fixing it now.

**Fix:** Replace `JetBrains_Mono` with `IBM_Plex_Mono` in Steps 3, 5, and 6. Update the import name to `IBM_Plex_Mono` from `next/font/google`, update the CSS variable to `--font-mono`, and update the dev-dep install accordingly. `IBM_Plex_Mono` is available on Google Fonts and works identically in `next/font/google`.

### Low — Nice to Fix

#### L-001: AC-005 verification command checks `0F7B5A` but plan lowercases hex values (Clarity)

**Issue:** The plan's `globals.css` content uses lowercase hex (`#0f7b5a`, `#1b3a8c`, etc.) but the AC-005 verification command does `grep -c "0F7B5A"` (uppercase). The grep will return 0, causing a false fail.

**Fix:** Use `grep -ic "0f7b5a"` (case-insensitive) or change the CSS to uppercase to match the architecture doc (which uses uppercase `#0F7B5A`). The architecture doc's palette block uses uppercase throughout.

---

#### L-002: `--font-mono` CSS variable name shadowed by `JetBrains_Mono` variable name (Clarity)

**Issue:** In Step 5, the plan declares `const jetbrainsMono = JetBrains_Mono({ variable: '--font-mono', ... })`. In Step 6 `globals.css`, the `@theme` block then maps `--font-mono: var(--font-mono), ui-monospace, monospace`. This creates a circular self-reference: the CSS variable resolves to itself. The `next/font` injection creates a scoped class variable (e.g. `--font-mono` on the `<html>` element), but the `@theme` token with the same name may not inherit it in all Tailwind v4 contexts.

**Fix:** The font CSS variable injected by `next/font` should use a distinct name that the `@theme` token then maps. Recommend: `variable: '--font-jetbrains'` (or `'--font-ibm-plex'` after the H-001 fix) in the font config, then `--font-mono: var(--font-ibm-plex), ui-monospace, monospace` in `@theme`. This is also how `--font-fraunces` and `--font-inter` are handled in the same plan — consistent with the pattern it already uses correctly for the other two fonts.
