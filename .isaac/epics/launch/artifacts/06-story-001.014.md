# Story 001.014: Push-back endpoint + modal

## Description

Build the `/api/push-back` feedback endpoint and the `<PushBackModal>` opened from the `[ push back ]` CTA on every Fieldwork card. Push-back submissions land in Vercel Blob as a JSON append log for Maria to review. Rate-limited, validated, honeypot-protected.

## Acceptance Criteria

- [ ] AC-001: `[ push back ]` CTA on `<FieldworkCard>` and `<FieldworkArticle>` opens `<PushBackModal>`
- [ ] AC-002: Modal fields: message (required, 10-2000 chars), name (optional, 0-80 chars), context (hidden, auto-populated with the Fieldwork slug being pushed back on)
- [ ] AC-003: Honeypot field (`<input type="text" name="website" style="display:none">`) — server rejects non-empty submissions as bot
- [ ] AC-004: Client-side validation with Zod; same schema reused server-side
- [ ] AC-005: `/api/push-back` POST accepts validated payload, rate-limits via Upstash (5 per hour per IP), writes to `push-back/{date}.jsonl` in Vercel Blob (one JSONL file per day, appended)
- [ ] AC-006: No email auto-forward in v1 — Maria reviews Blob log manually. Document the review process in `docs/push-back-review.md`.
- [ ] AC-007: Privacy notice visible in modal: *"We store what you send until Maria has reviewed it. We don't share it. No email, no newsletter — ever."* (mitigates PRV-001)
- [ ] AC-008: Submissions stored as **plain text** — no HTML, no Markdown rendering anywhere (mitigates SEC-006)
- [ ] AC-009: Success state: modal shows *"Received. Thanks for the argument."* — no redirect, user stays in context
- [ ] AC-010: Error states: 429 rate-limited (friendly message), 400 validation (highlight field), 500 upstream (generic apology)

## Technical Tasks

1. Build `src/app/api/push-back/route.ts` — edge runtime, Zod validation, rate limit (reusing infra from 001.011), Blob write
2. Build `src/components/PushBackButton.tsx` (client) — already referenced in 001.004, now wire to open modal
3. Build `src/components/PushBackModal.tsx` (client) — form, validation, submit, states
4. Build `src/lib/push-back/storage.ts` — helper: append JSON line to `push-back/{YYYY-MM-DD}.jsonl` in Blob
5. Build `src/lib/push-back/schema.ts` — Zod schemas (shared client + server)
6. Create `docs/push-back-review.md` — how Maria reads the Blob log (Vercel dashboard path, structure of JSONL)

## Files

- **MODIFY**: `src/components/PushBackButton.tsx` (wire open handler), `src/components/FieldworkCard.tsx` (pass slug context), `src/components/FieldworkArticle.tsx` (pass slug context)

## Scope rules

- Do NOT build an admin UI for reviewing submissions. Maria reads Blob files directly in v1.
- Do NOT email-notify on submission — privacy-respecting quiet feedback loop
- Do NOT collect IP addresses in the stored payload — rate limit uses IP but doesn't persist it
- Do NOT add captcha — honeypot + rate limit sufficient for v1 scale

## Risk mitigations (from 04-assessment.md)

- **SEC-006** push-back abuse → AC-003 (honeypot), AC-005 (rate limit), AC-008 (no rendering path)
- **PRV-001** GDPR → AC-007 (notice), AC-006 (retention policy documented)
