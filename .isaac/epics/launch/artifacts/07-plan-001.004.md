## Design decisions locked upfront

### 1. MDX compilation strategy — ship a `<MdxBody>` renderer

001.003 deferred MDX body compilation to render-time (`readMdxFile` returns `body: string`). This story operationalises that: a small `<MdxBody source={body} components={{...}}/>` server component that uses `@mdx-js/mdx`'s `compile()` + `MDXProvider`-equivalent approach.

**But — a simpler path exists for Next 15:** use `next-mdx-remote-client` or hand-rolled compile. Simpler still: since our content lives in git not remote, and build-time compilation is acceptable, we can use `@mdx-js/mdx` directly to compile at render time with `evaluate()` in a server component.

**Chosen path: `next-mdx-remote`-free hand-roll via `@mdx-js/mdx` `evaluate()`.**

```typescript
import { evaluate } from '@mdx-js/mdx';
import * as runtime from 'react/jsx-runtime';
import remarkGfm from 'remark-gfm';

const { default: Body } = await evaluate(source, {
  ...runtime,
  remarkPlugins: [remarkGfm],
});
```

`@mdx-js/mdx` is already installed as a transitive dep of `@mdx-js/loader` (v3.1.1 present). If it's not directly listed in `package.json`, add it. Verify before starting.

**Alternative rejected:** `next-mdx-remote` adds ~30KB for what we can do in 15 lines. `@mdx-js/mdx` is already in the tree.

### 2. Accent-colour system — CSS custom property on article root

Per AC-004: each piece sets `--color-accent` on its article root. Frontmatter `accent` wins if explicit; deterministic fallback based on `id % 5` ordered `[emerald, sapphire, ruby, topaz, amethyst]`.

A small `src/lib/design/accent.ts` helper exports:
- `ACCENT_ORDER: AccentToken[]` — the five-jewel rotation order
- `accentFor(piece: { id: number; frontmatter: { accent?: AccentToken } }): AccentToken`
- `accentVar(token: AccentToken): string` — returns `var(--color-${token})`

Tailwind v4 picks up new utilities automatically from `@theme` tokens. We already have `text-emerald`, `text-sapphire`, etc. What we need additionally is a way to drive `text-accent` / `bg-accent` / `border-accent` off `--color-accent`. Tailwind v4's arbitrary-value syntax handles this: `style={{ '--color-accent': 'var(--color-emerald)' }}` on the article root + `text-[color:var(--color-accent)]` usages inside.

**Chosen implementation:**
- `<FieldworkArticle>` and `<FieldworkCard>` wrap their roots in a `<div>` with inline style `{ '--color-accent': 'var(--color-emerald)' }` (derived).
- Utility CSS classes `text-accent` / `border-accent` / `bg-accent` added to `globals.css` as custom utilities:
  ```css
  @utility text-accent { color: var(--color-accent); }
  @utility border-accent { border-color: var(--color-accent); }
  @utility bg-accent { background-color: var(--color-accent); }
  ```
  (Tailwind v4 `@utility` directive — supported since 4.0.)

### 3. Route: `generateStaticParams`

For the `/fieldwork/[slug]` dynamic route:
- `generateStaticParams` reads `getAllFieldwork()` (no status filter — SSG retired pieces too per AC-003)
- Each piece's slug is emitted
- In `page.tsx`: call `getFieldworkBySlug(slug)`, render `<FieldworkArticle>`, or `notFound()` on null

Empty-content-dir case: `generateStaticParams` returns `[]`; Next 15 does NOT fail build on empty static params for a dynamic segment — confirmed by reading Next docs. Story AC-003 requires retired pieces to be SSG too with a *"retired — see archive"* banner.

### 4. Homepage — render Fieldwork cards

Update `src/app/page.tsx` (currently minimal placeholder) to:
- Fetch `getAllFieldwork({ status: 'in-rotation' })`
- Render `<BioLine />` (unchanged) + a grid of `<FieldworkCard>` for each in-rotation piece
- If empty (pre-001.005), fall back to a graceful "fieldwork arriving" note — preserves current ship-before-content-exists behaviour

### 5. Video placeholder

Per AC scope rules: placeholder only, no actual video. Create `<VideoLoopPlaceholder>` — server component, renders a rectangle at correct aspect ratio with an ink-on-paper "—video header—" label; 001.013 deletes and replaces.

### 6. `[ push back ]` / `[ watch ]` CTAs

Per scope rules: buttons exist, are no-ops. We ship them as plain `<button>` elements with `type="button"` and a `data-fieldwork-cta="push-back"` attribute (so 001.014 can select them for wiring) — no `onClick` at all, to avoid making the card a client component prematurely. **Explicitly:** this is NOT a `'use client'` island yet. Card remains a server component.

### 7. Metadata formatting

`date-fns` via `format(new Date(published), 'd MMM yyyy')` for "14 Apr 2026" style. UTC-anchor the Date construction to dodge the 001.002 timezone-test trap: `new Date(\`${published}T00:00:00Z\`)` + `formatInTimeZone` from `date-fns-tz` — but `date-fns-tz` isn't installed. Alternative: parse with `parseISO` + format in UTC via custom formatter, or just accept the timezone ambiguity since `YYYY-MM-DDT00:00:00Z` parses as UTC-midnight and formatting with `format()` (not `formatInTimeZone`) uses local tz — which on Vercel CI/prod (UTC) renders correctly.

**Chosen:** use `format(new Date(\`${published}T00:00:00Z\`), 'd MMM yyyy')`. Safe on Vercel (UTC) + developer machines UTC+0 (UK). Accept the UTC-1 display risk as trivially low for a UK-first site.

Also include a semantic `<time datetime={published}>` element wrapping the formatted date (AC-009).

```typescript
import type { AccentToken } from '@/lib/content/types';

export const ACCENT_ORDER: readonly AccentToken[] = [
  'emerald',
  'sapphire',
  'ruby',
  'topaz',
  'amethyst',
] as const;

/** Deterministic accent for a piece — explicit frontmatter first, id % 5 fallback. */
export function accentFor(
  piece: { frontmatter: { accent?: AccentToken; id: number } },
): AccentToken {
  return piece.frontmatter.accent ?? ACCENT_ORDER[piece.frontmatter.id % ACCENT_ORDER.length];
}

/** CSS `var()` reference for a jewel token — lets consumers set --color-accent. */
export function accentVar(token: AccentToken): string {
  return `var(--color-${token})`;
}
```

### Step 7 · Create `<MdxBody>` renderer

**File:** `src/components/MdxBody.tsx` (server)

```typescript
import { evaluate } from '@mdx-js/mdx';
import * as runtime from 'react/jsx-runtime';
import remarkGfm from 'remark-gfm';
import { PullQuote } from './PullQuote';
import { Aside } from './Aside';
import { FigureCaption } from './FigureCaption';

interface MdxBodyProps {
  source: string;
}

const components = {
  PullQuote,
  Aside,
  FigureCaption,
};

/**
 * Compiles and renders MDX content using @mdx-js/mdx evaluate().
 * Server-only — invoked during SSG in the route handler.
 */
export async function MdxBody({ source }: MdxBodyProps) {
  const { default: Body } = await evaluate(source, {
    ...runtime,
    remarkPlugins: [remarkGfm],
  });
  return <Body components={components} />;
}
```

**Type note:** `@mdx-js/mdx`'s `evaluate` signature accepts the jsx-runtime object. Types are sometimes looser than strict mode likes; if TS complains, cast: `...(runtime as any)` with an eslint-disable note. Default to trying the clean cast first.

import { describe, it, expect } from 'vitest';
import { accentFor, accentVar, ACCENT_ORDER } from '../accent';

const base = {
  body: '',
  filePath: 'x',
} as const;

describe('accentFor', () => {
  it('uses explicit frontmatter accent when present', () => {
    expect(
      accentFor({ ...base, frontmatter: { id: 3, accent: 'ruby' } } as never),
    ).toBe('ruby');
  });
  it('falls back to id % 5 when accent absent', () => {
    for (let i = 0; i < 10; i++) {
      expect(
        accentFor({ ...base, frontmatter: { id: i } } as never),
      ).toBe(ACCENT_ORDER[i % ACCENT_ORDER.length]);
    }
  });
  it('rotates deterministically (same id → same accent across calls)', () => {
    const a = accentFor({ ...base, frontmatter: { id: 7 } } as never);
    const b = accentFor({ ...base, frontmatter: { id: 7 } } as never);
    expect(a).toBe(b);
  });
});

describe('accentVar', () => {
  it('emits var(--color-<token>)', () => {
    expect(accentVar('emerald')).toBe('var(--color-emerald)');
    expect(accentVar('amethyst')).toBe('var(--color-amethyst)');
  });
});

```typescript
import { describe, it, expect } from 'vitest';
import { render } from '@testing-library/react';
import { MdxBody } from '../MdxBody';

describe('<MdxBody>', () => {
  it('compiles simple markdown into HTML', async () => {
    // Note: MdxBody is an async server component; vitest runs this in node env
    // because we're importing async fns. Just invoke and render the returned node.
    const node = await MdxBody({ source: '# Hello\n\nBody.' });
    const { container } = render(node);
    expect(container.querySelector('h1')).toHaveTextContent('Hello');
    expect(container.querySelector('p')).toHaveTextContent('Body.');
  });
  it('renders <PullQuote> JSX-in-MDX correctly', async () => {
    const node = await MdxBody({
      source: '<PullQuote>key quote</PullQuote>',
    });
    const { container } = render(node);
    expect(container.querySelector('blockquote')).toHaveTextContent('key quote');
  });
});
```

**Note:** MDX compilation in Node is stable; `evaluate()` from `@mdx-js/mdx` returns `{ default: Component }`. If either of these tests fails with a hydration warning inside jsdom, that's fine for the *compile* path — we're checking rendered output, not hydration.
