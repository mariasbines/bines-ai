**H-001 · `@vitejs/plugin-react` absent from `node_modules` — first test run will fail to compile JSX**

### Medium (2)

**M-001 · `CurrentlyStrip` test expects `Intl.DateTimeFormat` output; story test plan says "formats date via `date-fns`"**

- Severity: MEDIUM
- File: `06-story-001.002.md` line 43 vs `07-plan-001.002.md` Step 9 + Step 14
- Details: The story's test plan states *"Unit: `<CurrentlyStrip>` renders props correctly, formats date via `date-fns`"*. The plan's implementation uses `Intl.DateTimeFormat('en-GB', ...)` directly — not `date-fns`. The plan's test in `CurrentlyStrip.test.tsx` asserts against the `Intl` output (`"22 Apr 2026"`), which matches the implementation. `date-fns` is already installed but is not used in this component.
- Impact: Minor — the story's test-plan wording is aspirational; the plan is internally consistent. However if an executor or reviewer reads both documents they will see a mismatch. Testability is unaffected.
- Suggested fix: Either add a one-line comment in `CurrentlyStrip.tsx` noting the choice, or update the story's test-plan bullet to read "formats date via `Intl.DateTimeFormat`". Not a blocker.

**M-002 · UTC midnight timezone edge case in `CurrentlyStrip` test**

- Severity: MEDIUM
- File: `src/components/__tests__/CurrentlyStrip.test.tsx` (Step 14)
- Details: Test fixture uses `new Date('2026-04-22T00:00:00Z')`. `Intl.DateTimeFormat('en-GB', ...)` formats in local timezone. In UTC+0 this renders `"22 Apr 2026"` (verified). In a CI environment running west of UTC (e.g., UTC-5), UTC midnight becomes 21 Apr, causing the assertion `/updated 22 Apr 2026/` to fail.
- Impact: Test could produce a false negative on some CI runners. Vercel CI runs on UTC — likely safe in practice, but fragile.
- Suggested fix: In the test, construct the date with `new Date(2026, 3, 22)` (local date constructor) rather than an ISO UTC string, or pass `timeZone: 'UTC'` to the formatter. Low-effort fix for a real edge.

### Low (3)

**L-001 · `PageShell` imports `React.ReactNode` without importing React — TS strictness note**

- Severity: LOW
- File: `src/components/PageShell.tsx` Step 11
- Details: `interface PageShellProps { children: React.ReactNode }` requires `React` to be in scope. In React 19 + TypeScript 5 with the automatic JSX transform, `React` does not need to be imported for JSX — but it does need to be imported (or the interface rewritten as `import type { ReactNode } from 'react'; children: ReactNode`) for the type reference. With `skipLibCheck: true` this often passes anyway, but strict TypeScript may flag it.
- Suggested fix: Add `import type { ReactNode } from 'react';` and use `ReactNode` directly, or add `import React from 'react';`. One line.

**L-002 · No test for `<PageShell>` composition**

The risk is that the `layout.tsx` modification (Step 12) is untested at unit level. Visual smoke at `localhost:3000` is the only verification.
- Suggested fix: Acceptable as-is per the plan's stated rationale. Call-out for awareness only.

**L-003 · `SITE_STATS_PLACEHOLDER.updated` is a module-level `new Date()` — will be stale the moment 001.008 ships**
