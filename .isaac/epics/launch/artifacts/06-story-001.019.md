# Story 001.019: CI + automated Claude PR review (Tier 3b)

## Description

Stand up the regression-protection layer: GitHub Actions for typecheck / lint / test / build on every PR, plus an automated Claude review that posts comments on PRs targeting `master`. Enable branch protection on `master` so merges are only possible once CI is green and a review is approved.

This story is **inserted after 001.001 and blocks 001.003** — it must be in place before the first 🔴 Full-TDD story (content pipeline) lands, so CI actually catches the tests. The automated Claude review is specifically Tier 3b from the review-strategy discussion: always-on, model-driven PR review that complements (doesn't replace) the ISAAC grade + validate phases.

## Acceptance Criteria

- [ ] AC-001: `.github/workflows/ci.yml` exists — triggers on PR to `master` and push to `dev`; jobs run in this order: `pnpm install --frozen-lockfile` → `pnpm typecheck` → `pnpm lint` → `pnpm test` → `pnpm build`; any non-zero exit fails the workflow
- [ ] AC-002: `.github/workflows/claude-review.yml` exists — triggers on PR `opened` and `synchronize` events targeting `master`; calls a Node script that invokes the Anthropic API with the PR diff + story context and posts the review as a PR comment
- [ ] AC-003: `.github/scripts/review-pr.mjs` exists — computes the diff via `gh pr diff`, extracts the story ID from the PR title or body (convention: `[001.XXX]` prefix or a body line `story: 001.XXX`), loads the referenced story file plus `CLAUDE.md` + `03-architecture.md` as context, calls Claude Sonnet 4.6, posts a single review comment
- [ ] AC-004: Review prompt (hardcoded in script, separate from ISAAC principles) covers: correctness vs story ACs, scope adherence vs story "Files to Modify", security red flags (hardcoded secrets, XSS paths, prompt-injection gaps for chat stories), regression risk ("what existing behaviour might this break"), voice drift (for content / system-prompt changes), typographic rigour at output (specific file:line citations, not vague advice)
- [ ] AC-005: Review output shape is consistent: `## Must-fix` / `## Suggestions` / `## Nothing to flag` sections; max ~400 words per review; prefix the comment with a visible Claude-review-bot header so the automated review is distinguishable from human comments
- [ ] AC-006: GitHub secret `ANTHROPIC_API_KEY` set on the repo (Maria action — `gh secret set ANTHROPIC_API_KEY` or dashboard)
- [ ] AC-007: Branch protection rule on `master` (Maria action via GitHub settings): require CI workflow green, require at least 1 approving review, require conversations resolved, disallow force-push, disallow deletion
- [ ] AC-008: `.github/pull_request_template.md` exists with a short checklist: story referenced (`[001.XXX]`), ACs addressed, tests added/updated, scope rules respected, voice check done if applicable, preview URL reviewed
- [ ] AC-009: `CLAUDE.md` updated with a short section on the PR review flow — when to trust automated review vs run `/security-review` manually (chat / push-back / env-var stories specifically)
- [ ] AC-010: Dry-run verified: open a trivial PR from a test branch, observe CI run to completion and Claude review comment posted within 60s of PR open

## Scope rules

- Do NOT add additional workflows beyond CI + Claude review (no auto-deploy-on-merge — Vercel handles that; no release automation; no changelog generator)
- Do NOT enable Claude review on `dev` pushes — only on PRs to `master` (would be noisy and expensive for every wip commit)
- Do NOT rate-limit or cache Claude reviews — fresh review per PR sync event; if we see cost issues, revisit
- Do NOT store review output anywhere beyond the PR comment — ephemeral by design
- The review bot is advisory, not authoritative — it posts comments, never approves/blocks

## Risk mitigations

- **Regression protection (Maria's stated concern)**: CI blocks merges on failing tests; Claude review flags potential regressions in the PR comment
- **SEC-005 (API key exposure)**: `ANTHROPIC_API_KEY` in GitHub secrets, never echoed in logs, never checked into the repo
- **Review-bot mistakes**: output explicitly framed as advisory; human approval still required; `CLAUDE.md` notes how to dispute a bot finding

## Test plan (🟡 Critical Path)

- Static: YAML workflows validate via `actionlint` (or GitHub's built-in validator) before push
- Integration: test PR opens against `master`, CI runs, review comment appears within 60s
- Failure paths:
  - Trigger a deliberate `pnpm test` failure on a test PR — CI fails, merge is blocked
  - Submit a PR with a hardcoded secret — Claude review should flag it as must-fix
  - Submit a PR that violates story scope — Claude review should flag scope drift
- Cost sanity: after 5 test PRs, check Anthropic billing console; document typical cost-per-PR in `CLAUDE.md`
