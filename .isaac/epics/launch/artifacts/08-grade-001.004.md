### High (1)

**H-001 · `@mdx-js/mdx` must be declared in `dependencies`; currently a transitive dep only**

- Severity: HIGH
- File: `package.json`
- Details: Verified via `pnpm list @mdx-js/mdx --depth=0` — the package is NOT directly declared in `package.json`. It's pulled in transitively by `@mdx-js/loader@3.1.1`. In development + Vercel builds this usually resolves because of pnpm's hoisting (`node_modules/.pnpm/@mdx-js+mdx@3.1.1/` exists). HOWEVER: explicitly importing a package that's only a transitive dep is fragile — version bumps of `@mdx-js/loader` could orphan it. Plan Step 1 calls this out and proposes `pnpm add @mdx-js/mdx` as the fix.
- Impact: Medium-high — skipping the install would let dev work locally but could break Vercel preview / production builds under pnpm's stricter resolution (`pnpm install --frozen-lockfile` in CI).
- Suggested fix: Plan already handles it in Step 1. Executor must not skip.

### Medium (2)

**M-001 · `vi.mock` placement in `FieldworkArticle.test.tsx`**

- Severity: MEDIUM
- File: `src/components/__tests__/FieldworkArticle.test.tsx` (proposed Step 12)
- Details: The plan shows `vi.mock(...)` at the top followed by `import { vi }` AFTER it. `vi.mock` is hoisted by Vitest, so this works — but the plan text explicitly notes "Reorder in actual file." The executor must follow that note. If the import ordering is left as shown in the plan text without reordering, ESLint may complain (import-after-statement).
- Impact: Low — one file, one line move.
- Suggested fix: Put `import { vi } from 'vitest';` at the top of imports alongside the other vitest import, then `vi.mock(...)` — executor must follow the plan's own text ("Important: ... Reorder in actual file.").

**M-002 · `dt`/`dd` grid layout — watch out for span consistency**

- Severity: MEDIUM
- File: `src/components/Metadata.tsx` (proposed Step 4)
- Details: The plan uses `<dl>` with `grid grid-cols-[auto,1fr]`. Each `<dt>/<dd>` pair occupies one row. For the "media" row, the `<dd>` concatenates the watch + read string. This renders correctly in modern browsers.
- Impact: None functionally; called out for awareness — screen readers may announce the dl as a definition list which is semantically correct.
- Suggested fix: Accept as-is.

### Low (4)

**L-001 · `formatInTimeZone` vs naive `format` — timezone drift on non-UTC runners**

- Severity: LOW
- File: `src/components/Metadata.tsx` Step 4
- Details: Plan acknowledges this tradeoff explicitly. `new Date('2026-04-14T00:00:00Z')` anchors UTC-midnight; `format(d, 'd MMM yyyy')` formats in local tz. On UTC-0 runners (Vercel, UK dev), renders "14 Apr 2026". On UTC-5 (a US developer), renders "13 Apr 2026". Personal site targeted at UK/Europe — acceptable risk.
- Impact: Low — test could fail on a US developer's laptop. CI (GitHub Actions Ubuntu = UTC) is safe.
- Suggested fix: Keep as planned; if a non-UTC dev hits this, swap to UTC-aware formatting with a small helper.

**L-002 · Empty static-params handling on `/fieldwork/[slug]`**

- Severity: LOW
- File: `src/app/fieldwork/[slug]/page.tsx` Step 10
- Details: Plan asserts Next 15 handles `generateStaticParams()` returning `[]` without error. Verified from existing epic context (001.001 foundation); this was the ship-without-content behaviour enforced in earlier stories.
- Suggested fix: Accept — plan already calls this out.

**L-003 · `<MdxBody>` is async server component; RTL can't render async components directly**

- Severity: LOW
- File: `src/components/__tests__/MdxBody.test.tsx` (proposed Step 12)
- Details: Plan's workaround is to `await MdxBody({ source })` directly, treating it as a function, and then render the returned JSX node. This is the canonical Vitest pattern for async server components (confirmed from Next.js + Vitest cookbook examples).
- Suggested fix: Accept as planned. If Vitest + React 19 unstable flags cause issues, swap to using `react-dom/server` renderToStaticMarkup for a simpler assertion.

**L-004 · No test for `<FigureCaption>` or `<Aside>`**

- Severity: LOW
- File: Test coverage (Step 12)
- Details: Plan tests `<PullQuote>` explicitly but not `<FigureCaption>` or `<Aside>`. Both are trivial (children + semantic wrapper); risk is very low.
- Suggested fix: Acceptable — `PullQuote` test acts as the pattern exemplar; these two are structurally identical.
