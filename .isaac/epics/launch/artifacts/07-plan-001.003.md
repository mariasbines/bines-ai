# Implementation Plan: 001.003 — Content pipeline (Zod schemas + MDX loaders)

## Overview

Build the MDX content pipeline — Zod schemas for frontmatter validation, `gray-matter` for frontmatter parsing, and typed server-only loaders that read MDX files from `content/` and return parsed, validated content objects. This is the data layer everything downstream depends on (Fieldwork rendering 001.004, Postcard rendering 001.006, /now + /taste 001.008, archive 001.009, changed-my-mind 001.010, RSS/sitemap 001.015). Break this and everything breaks.

The story is 🔴 Full TDD — schemas and loaders must be tested exhaustively before anything consumes them.

## Design decisions locked upfront

### 1. MDX compilation strategy — `@next/mdx` + on-demand helper

`@next/mdx` is already wired in `next.config.ts` with `remark-gfm`. It compiles `.mdx` files imported under `src/app/...` into React components. However, our content lives under `content/` (repo root, NOT under `src/app/`) — so direct `@next/mdx` route-based compilation doesn't apply. Instead:

- **Loaders read raw MDX text from disk** via `fs.readFile` (server-only).
- **Frontmatter parsing** via `gray-matter`.
- **MDX body compilation** is NOT done in the loader itself in this story. Reason: Next.js will compile MDX when the body is rendered (story 001.004 wires this via dynamic import). In 001.003 we expose the **raw body string** plus a `compile()` helper that callers use when rendering. This keeps 001.003 boundary clean: loaders handle data + validation; compilation is an orthogonal concern.

This matches what the story says in AC-003: *"returns a single piece with compiled MDX content (using `@next/mdx`)"*. We interpret *"compiled MDX content"* as **the MDX loader exports a raw body that is ready to be passed to the MDX renderer**. 001.004 handles the actual render path. (Alternative — pre-compile in loader via `@mdx-js/mdx` and return a JSX tree — rejected: adds a heavy dep path in this story and couples build to render.)

**Deferred to 001.004:** actual render via `MDXRemote` / import; the loader interface exports `body: string` and a `bodySource(slug)` helper for Next's MDX page-file pattern.

### 2. Schema: discriminated unions for Fieldwork status

Story 001.010 extends `FieldworkFrontmatter` into a discriminated union on `status` (the `"changed-my-mind"` variant requires `supersedes`/`originalPosition`/`newPosition`). **001.003 ships the base schema as a plain object** with `status` accepting all four enum values, and `supersedes`/`originalPosition`/`newPosition` as OPTIONAL fields at the top level. This is Codex-verifiably sufficient for now: loaders validate frontmatter, and the cross-reference validation (must-point-at-real-slug) is deferred to 001.010 which has that AC explicitly.

**Rationale:** if 001.003 shipped the full discriminated union now, 001.004 content (which only uses `in-rotation`) validates fine and 001.010 tightens the schema later. Keeping fields flat-optional in 001.003 avoids a churny refactor in 001.010 that would touch loaders + tests for a story that's already doing schema work. Cost: two fields present on types that won't use them until 001.010. Trade accepted.

### 3. Server-only enforcement

Each loader file starts with `import 'server-only'`. This is the Next 15 recommended pattern; any accidental client import throws a clear compile-time error. We also set a runtime `process.env.NEXT_RUNTIME` assertion is NOT needed (the `server-only` import is sufficient and canonical).

### 4. Testing strategy — fixtures in tmp dirs

Loaders take a `contentDir` parameter with a default of `path.resolve(process.cwd(), 'content')`. Tests inject `contentDir` pointing at a per-test `os.tmpdir()` directory seeded with fixture MDX files. This avoids coupling tests to the real `content/` tree (which will be empty until 001.005 / 001.007 seed it).

Pattern:

```typescript
const tmp = await fs.mkdtemp(path.join(os.tmpdir(), 'bines-content-'));
await fs.mkdir(path.join(tmp, 'fieldwork'));
await fs.writeFile(path.join(tmp, 'fieldwork', '01-foo.mdx'), '---\n...\n---\nbody');
// ... exercise loader
// cleanup: await fs.rm(tmp, { recursive: true, force: true });
```

### 5. `getCurrentlyLine` + `getSiteStats` — compose other loaders

`getSiteStats()` calls `getAllFieldwork()`, `getAllPostcards()`, and `getFieldworkByStatus('changed-my-mind')` and returns counts + the max `updated` timestamp across the tree. `getCurrentlyLine()` calls `getNow()` and returns its `currently` field. Both are pure composition over the base loaders.

### Step 3 · Create `src/lib/content/mdx.ts` (AC-005)

**File:** `src/lib/content/mdx.ts` (CREATE) — shared parsing helper used by every loader.

```typescript
import 'server-only';

import matter from 'gray-matter';
import { promises as fs } from 'node:fs';
import path from 'node:path';
import type { ZodType } from 'zod';

export class ContentValidationError extends Error {
  constructor(
    public readonly filePath: string,
    public readonly zodIssues: unknown,
    message: string,
  ) {
    super(message);
    this.name = 'ContentValidationError';
  }
}

/**
 * Read an MDX file and parse its frontmatter against the given Zod schema.
 *
 * Throws `ContentValidationError` with file path + Zod issues if frontmatter
 * fails validation. Missing files throw `ENOENT`, caller decides whether
 * to return null or propagate.
 */
export async function readMdxFile<T>(
  filePath: string,
  schema: ZodType<T>,
): Promise<{ frontmatter: T; body: string; filePath: string }> {
  const raw = await fs.readFile(filePath, 'utf8');
  const parsed = matter(raw);

  const result = schema.safeParse(parsed.data);
  if (!result.success) {
    const summary = result.error.issues
      .map((i) => `  - ${i.path.join('.') || '(root)'}: ${i.message}`)
      .join('\n');
    throw new ContentValidationError(
      filePath,
      result.error.issues,
      `Invalid frontmatter in ${filePath}:\n${summary}`,
    );
  }

  return {
    frontmatter: result.data,
    body: parsed.content,
    filePath,
  };
}

/**
 * List all `.mdx` files in a directory (non-recursive).
 * Returns absolute paths sorted alphabetically for determinism.
 */
export async function listMdxFiles(dir: string): Promise<string[]> {
  let entries;
  try {
    entries = await fs.readdir(dir, { withFileTypes: true });
  } catch (err) {
    if ((err as NodeJS.ErrnoException).code === 'ENOENT') return [];
    throw err;
  }
  return entries
    .filter((e) => e.isFile() && e.name.endsWith('.mdx'))
    .map((e) => path.join(dir, e.name))
    .sort();
}
```

**Why `fs.readdir` with `withFileTypes`:** lets us filter to files-only, rejecting the `.gitkeep` marker and any nested directories.
**Why tolerate missing dir:** empty content trees (before 001.005 seeds) must return `[]`, not throw. Satisfies AC-002 edge case "handles empty dir".

### Step 5 · Create `src/lib/content/fieldwork.ts` (AC-002, AC-003)

**File:** `src/lib/content/fieldwork.ts` (CREATE)

```typescript
import 'server-only';

import { listMdxFiles, readMdxFile } from './mdx';
import { contentPaths, DEFAULT_CONTENT_ROOT } from './paths';
import { FIELDWORK_FRONTMATTER, type Fieldwork, type FieldworkStatus } from './types';

interface LoaderOptions {
  contentRoot?: string;
  /** Filter by status — e.g. only `in-rotation` pieces for homepage. */
  status?: FieldworkStatus;
}

/**
 * Read every `.mdx` under `content/fieldwork`, validate, return sorted
 * descending by `published`. Empty dir = [].
 */
export async function getAllFieldwork(options: LoaderOptions = {}): Promise<Fieldwork[]> {
  const { fieldworkDir } = contentPaths(options.contentRoot ?? DEFAULT_CONTENT_ROOT);
  const files = await listMdxFiles(fieldworkDir);

  const pieces = await Promise.all(
    files.map((f) => readMdxFile(f, FIELDWORK_FRONTMATTER)),
  );

  const filtered = options.status
    ? pieces.filter((p) => p.frontmatter.status === options.status)
    : pieces;

  return filtered.sort((a, b) =>
    b.frontmatter.published.localeCompare(a.frontmatter.published),
  );
}

/** Returns null if no piece matches `slug`. Never throws for missing slug. */
export async function getFieldworkBySlug(
  slug: string,
  options: Omit<LoaderOptions, 'status'> = {},
): Promise<Fieldwork | null> {
  const all = await getAllFieldwork({ contentRoot: options.contentRoot });
  return all.find((p) => p.frontmatter.slug === slug) ?? null;
}

/** Shortcut for a single status — used by /archive in 001.009. */
export async function getFieldworkByStatus(
  status: FieldworkStatus,
  options: Omit<LoaderOptions, 'status'> = {},
): Promise<Fieldwork[]> {
  return getAllFieldwork({ ...options, status });
}

/** For /archive grouping (001.009 consumer). Empty categories map to []. */
export async function getFieldworkGroupedByStatus(
  options: Omit<LoaderOptions, 'status'> = {},
): Promise<Record<FieldworkStatus, Fieldwork[]>> {
  const all = await getAllFieldwork({ contentRoot: options.contentRoot });
  const groups: Record<FieldworkStatus, Fieldwork[]> = {
    'in-rotation': [],
    'retired-still-right': [],
    'retired-evolved': [],
    'changed-my-mind': [],
  };
  for (const piece of all) {
    groups[piece.frontmatter.status].push(piece);
  }
  return groups;
}
```

**ISO date note:** `published` is `"YYYY-MM-DD"`. Lexicographic sort on that format IS correct chronological sort — no need to `new Date()` parse.

import { listMdxFiles, readMdxFile } from './mdx';
import { contentPaths, DEFAULT_CONTENT_ROOT } from './paths';
import { POSTCARD_FRONTMATTER, type Postcard } from './types';

### Step 9 · Create `src/lib/content/stats.ts` (AC-006)

**File:** `src/lib/content/stats.ts` (CREATE)

```typescript
import 'server-only';

import { getAllFieldwork } from './fieldwork';
import { getAllPostcards } from './postcards';
import { getNow } from './now';
import { DEFAULT_CONTENT_ROOT } from './paths';
import type { SiteStats } from './types';

interface LoaderOptions {
  contentRoot?: string;
}

/**
 * Aggregate counts for <CurrentlyStrip> + archive header.
 *
 * `updated` = max of:
 *   - /now `updated` if present
 *   - latest `published` across Fieldwork
 *   - latest `published` across Postcards
 *   - falls back to epoch 0 if the site is completely empty
 */
export async function getSiteStats(options: LoaderOptions = {}): Promise<SiteStats> {
  const root = options.contentRoot ?? DEFAULT_CONTENT_ROOT;
  const [fw, pc, now] = await Promise.all([
    getAllFieldwork({ contentRoot: root }),
    getAllPostcards({ contentRoot: root }),
    getNow({ contentRoot: root }),
  ]);

  const changedMyMindCount = fw.filter(
    (p) => p.frontmatter.status === 'changed-my-mind',
  ).length;

  const candidates: string[] = [];
  if (now) candidates.push(now.frontmatter.updated);
  if (fw[0]) candidates.push(fw[0].frontmatter.published);
  if (pc[0]) candidates.push(pc[0].frontmatter.published);

  const latestIso =
    candidates.length === 0
      ? '1970-01-01'
      : candidates.sort().reverse()[0];

  return {
    fieldworkCount: fw.length,
    postcardCount: pc.length,
    changedMyMindCount,
    updated: new Date(`${latestIso}T00:00:00Z`),
  };
}
```

**Why UTC-anchor on the date:** downstream `<CurrentlyStrip>` already formats with `Intl.DateTimeFormat('en-GB', ...)`. Anchoring to UTC avoids cross-timezone drift on CI (we hit this on 001.002 grade — M-002).

## Clarifying Questions

4. **MDX body — raw string or pre-compiled?** Plan returns raw `body: string`. 001.004 will wire the actual MDX render via Next's pattern.

   **Recommendation: raw string.** Cleanest boundary. Avoids pulling a second MDX compiler into this story.
