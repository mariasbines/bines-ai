# Implementation Plan: 001.014 — Push-back endpoint + modal

## Overview

Build `/api/push-back` feedback endpoint + `<PushBackModal>` opened from the `[ push back ]` CTA. Submissions land in Vercel Blob as a daily JSONL log. Honeypot-protected, rate-limited (reuses 001.011's Upstash infra), Zod-validated.

## Design decisions

### 1. Rate limit — new module or reuse

001.011's `rate-limit.ts` is chat-specific (10/10min + 50/day). Push-back needs its own scale (5/hr/IP per AC-005). Add a new `getPushBackRatelimit()` helper inside `src/lib/chat/rate-limit.ts` — keeps Upstash-setup code DRY.

Actually, push-back isn't "chat" — better to have a shared `src/lib/rate-limit.ts` that exports getters for each named limit. Alternative: put it in `src/lib/push-back/rate-limit.ts`. Push-back has its own module scope; the cleanest move is a local one in push-back/.

**Chosen:** `src/lib/push-back/rate-limit.ts` — local to push-back, follows the pattern from 001.011.

### 2. Blob storage shape

Daily JSONL: `push-back/{YYYY-MM-DD}.jsonl`. Each submission appended as one line:

```json
{"timestamp": "2026-04-22T12:34:56Z", "slug": "01-best-thing", "message": "...", "name": "Anon"}
```

Vercel Blob's API: `put(pathname, body, { access, contentType, addRandomSuffix: false })`. For append semantics, we GET the existing file (if any), concat, PUT back. Race conditions are acceptable at v1 scale (few submissions, infrequent concurrent writes).

Per AC-006, no IP address is persisted — rate-limit uses IP but doesn't write it.

### 3. Schema — shared client + server

`src/lib/push-back/schema.ts`:

```typescript
import { z } from 'zod';

export const PUSH_BACK = z.object({
  slug: z.string().min(1).max(200),
  message: z.string().min(10, 'at least 10 characters').max(2000, 'at most 2000'),
  name: z.string().max(80).optional().or(z.literal('')),
  // Honeypot — must be empty or absent
  website: z.string().max(0).optional(),
});
export type PushBack = z.infer<typeof PUSH_BACK>;
```

Honeypot enforcement: `z.string().max(0)` — non-empty strings fail validation.

### 4. Modal — native `<dialog>` (matches WatchDialog from 001.013)

Client component. Form with:
- Hidden `slug` (auto-populated from context)
- Textarea for message (10-2000 chars, client-side character count)
- Name (optional)
- Honeypot: `<input name="website" tabIndex={-1} aria-hidden style={display:none}>`
- Submit button

On submit: POST to `/api/push-back`, handle success (show confirmation, auto-close after 2s) or error (inline message).

### 5. Route `/api/push-back`

Edge runtime. Orchestrates: parse → validate (Zod) → rate limit → Blob append → 200 or error.

### 6. Wiring — enable [ push back ] button

Replace the disabled button in `FieldworkCardCtas` with a button that opens `<PushBackModal slug={slug}>`. Same pattern as the watch CTA. Similarly, wire on `FieldworkArticle` so readers can push back from the article page too.

### Step 2 · Storage — `src/lib/push-back/storage.ts`

```typescript
import 'server-only';
import { list, put } from '@vercel/blob';
import type { PushBack } from './schema';

const JSONL_PREFIX = 'push-back/';

function dateKey(d: Date = new Date()): string {
  const iso = d.toISOString();
  return iso.slice(0, 10); // YYYY-MM-DD
}

export async function appendPushBack(entry: PushBack & { timestamp: string }): Promise<void> {
  const token = process.env.BLOB_READ_WRITE_TOKEN;
  if (!token) throw new Error('BLOB_READ_WRITE_TOKEN missing');
  const filename = `${JSONL_PREFIX}${dateKey()}.jsonl`;
  // Check if the file exists; Vercel Blob's `list` returns URLs
  const existing = await list({ prefix: filename, token });
  let body = JSON.stringify(entry) + '\n';
  if (existing.blobs.length > 0) {
    // Fetch existing content and prepend
    const res = await fetch(existing.blobs[0].url);
    if (res.ok) body = (await res.text()) + body;
  }
  await put(filename, body, {
    access: 'public', // Maria reads via Vercel dashboard; public Blob URL is fine
    contentType: 'application/x-ndjson',
    addRandomSuffix: false,
    token,
    allowOverwrite: true,
  });
}
```

`@vercel/blob` requires explicit token in edge runtime (no filesystem for credentials). `addRandomSuffix: false` + `allowOverwrite: true` to keep the filename stable.

### Step 3 · Rate limit — `src/lib/push-back/rate-limit.ts`

```typescript
import { Ratelimit } from '@upstash/ratelimit';
import { Redis } from '@upstash/redis';

let _rl: Ratelimit | null = null;

export function getPushBackRatelimit(): Ratelimit | null {
  if (_rl) return _rl;
  const url = process.env.UPSTASH_REDIS_REST_URL;
  const token = process.env.UPSTASH_REDIS_REST_TOKEN;
  if (!url || !token) return null;
  _rl = new Ratelimit({
    redis: new Redis({ url, token }),
    limiter: Ratelimit.slidingWindow(5, '1 h'), // 5 per hour per IP
    analytics: false,
    prefix: 'bines:pushback:1h',
  });
  return _rl;
}

export function __resetPushBackRatelimitForTests(): void {
  _rl = null;
}
```

## Clarifying Questions

1. **Blob access=public vs private** — Maria reads daily logs via Vercel dashboard. `public` Blob URLs are unguessable but discoverable to anyone with the URL. Since we're not collecting emails or PII, public-with-obscure-path is acceptable.
2. **Append via fetch+concat+put** — not atomic. Two concurrent requests could race. v1 scale is low enough to not matter; v2 can move to a dedicated queue or per-submission files.
