# Story 001.012: Chat interface (client) — /argue page

## Description

Build the client-side chat interface on `/argue`. Consumes the `/api/chat` streaming endpoint from 001.011. Renders streamed tokens progressively, distinguishes error states (rate-limited / service-down / unknown) in voice, shows the "AI not Maria" disclaimer, and is keyboard-navigable.

## Acceptance Criteria

- [ ] AC-001: `/argue` renders with `<PageShell>` chrome + the chat interface below a short intro ("argue with me — this is an AI trained in Maria's voice")
- [ ] AC-002: Chat UI: message list (user messages right-aligned, AI responses left-aligned in Maria's serif), text area, submit button labelled *"push back"* (nod to the Fieldwork CTA)
- [ ] AC-003: Messages render as **plain text** — newlines preserved via `white-space: pre-wrap`, no Markdown/HTML parsing (mitigates SEC-004)
- [ ] AC-004: Streaming: tokens appear progressively as the response streams; no jank, no layout shift
- [ ] AC-005: Conversation lives in component state only — refresh clears it. Short note under input: *"this conversation is not stored. what you type is sent to Anthropic."* (mitigates PRV-002)
- [ ] AC-006: Error states in-voice: 429 rate-limited ("ease up — we've had a lot of arguing today"), 5xx upstream ("Claude's having a moment, try again in a sec"), unknown network failure ("that didn't go through — try once more"). Copy lives in `src/lib/chat/error-messages.ts`.
- [ ] AC-007: Below the input, a small disclaimer block: *"You're talking to an AI trained on Maria's voice. It's not Maria. It gets things wrong sometimes — push back when it does."* (mitigates BRD-003)
- [ ] AC-008: Keyboard: input autofocuses, Enter submits (Shift-Enter for newline), Escape aborts in-flight request
- [ ] AC-009: `prefers-reduced-motion`: streaming appears instantly instead of token-by-token animation
- [ ] AC-010: Accessible: input has label, responses have `role="log"` + `aria-live="polite"`, error messages have `role="alert"`

## Technical Tasks

1. Build `src/app/argue/page.tsx` — server shell with intro + `<ChatInterface>`
2. Build `src/components/ChatInterface.tsx` (client component) — streaming fetch, state management, UI
3. Build `src/lib/chat/client.ts` — helper: POST to `/api/chat`, iterate streaming response
4. Build `src/lib/chat/error-messages.ts` — voice-appropriate error copy
5. Add `src/components/ChatMessage.tsx` — render a single message (user vs assistant)
6. Add `src/components/ChatInput.tsx` — textarea + submit button with loading state

## Scope rules

- Do NOT persist conversations anywhere — in-memory state only
- Do NOT render Markdown or HTML in responses — plain text only (SEC-004 defence)
- Do NOT build a sidebar or drawer — v1 is a dedicated page
- Do NOT add voice / avatar — v3 feature

## Risk mitigations (from 04-assessment.md)

- **SEC-004** output XSS → AC-003 (plain text rendering only)
- **BRD-003** reputation → AC-007 (disclaimer)
- **OPS-001** silent failures → AC-006 (error UX)
- **PRV-002** data residency → AC-005 (privacy note)
