### High (2)

**H-001 · Edge-runtime fs limitation — Fieldwork 01 one-shot loading strategy must actually work under edge**

- Severity: HIGH
- File: `src/lib/chat/fieldwork-01.ts` (proposed Step 1) vs route runtime
- Details: Plan Step 1 says "copy the body into the source as a TypeScript constant." This is correct — it works under edge runtime because no filesystem access is needed. The plan is internally consistent; the HIGH severity rating is for the importance of the strategy (not a defect).
- Verified: `fs.readFile` is unavailable on edge runtime (Next 15 docs confirmed). Our content loaders from 001.003 DO use fs and are correctly marked `server-only` — they cannot be imported by an edge route. The plan's approach (TS constant copy) bypasses this correctly.
- Suggested fix: Executor must NOT import from `@/lib/content/fieldwork` in the system-prompt module — that would break the edge build. Use only the inlined TS constant.
- Parity note: a comment in `fieldwork-01.ts` should explicitly call out the sync-with-MDX requirement.

**H-002 · `getAnthropicClient` throws when API key absent — route must catch and return 500, not crash module import**

- Severity: HIGH
- File: `src/lib/chat/anthropic.ts` + route Step 7
- Details: If `ANTHROPIC_API_KEY` is absent, the `getAnthropicClient()` call throws. The route's try/catch wraps the call, so the request returns 500 — but module-level code shouldn't crash. Plan is correct: `getAnthropicClient()` is invoked inside the POST handler, not at module load. Key resolution is lazy.
- Verified: Plan places `getAnthropicClient()` inside the POST handler, not at module top-level. Correct.
- Suggested fix: None — the plan is right. This was a potential trap, avoided.

### Medium (3)

**M-001 · Agent-guard doesn't inject detection into the prompt when `isLikelyAgent === true`**

- Severity: MEDIUM
- File: Plan Step 7 (agent-guard usage)
- Details: Plan detects-and-logs only (`console.log` signals). Doesn't inject an "untrusted content flag" note into the messages before passing to Anthropic. For SEC-001 mitigation, detect-and-log is the minimum; detect-and-augment is stronger. Moss pattern at `~/synapsedx-main/...` does log-only; bines.ai matches that.
- Suggested fix: Accept for v1. If vibe-check + real-traffic logs show injection attempts slipping through, upgrade in a follow-up story to insert a system-note like `<injection_signals>${guard.signals.join(',')}</injection_signals>` before the last user message.

**M-002 · Rate-limit IP extraction from `x-forwarded-for` — spoofing concern**

- Severity: MEDIUM
- File: Plan Step 7 `getIp`
- Details: Plan reads `x-forwarded-for` header which is client-trusted. Attackers can spoof this to bypass rate limiting. On Vercel, the first IP in `x-forwarded-for` is the real client IP (Vercel injects it); downstream headers can have spoofed additions. The plan uses `xf.split(',')[0]` — correct for Vercel's format.
- Suggested fix: Accept — matches Vercel's documented behaviour. Note in a code comment that this assumes Vercel's IP injection; non-Vercel deployments may need different extraction.

**M-003 · `anthropic.messages.stream` throws path vs stream-error path — how does SDK surface mid-stream errors?**

- Severity: MEDIUM
- File: Plan Step 7 route
- Details: Plan's try/catch wraps the `client.messages.stream(...)` invocation. But streaming errors can occur mid-stream (rate-limited by Anthropic, network drop). The plan's return of `stream.toReadableStream()` hands the stream to Next which pipes it to the client. Mid-stream errors become client-visible as truncated streams, not as HTTP 500s.
- Impact: Medium. Client sees a truncated response. Client-side code in 001.012 needs to handle this (graceful "that didn't go through — try once more").
- Suggested fix: Document in a code comment that mid-stream errors surface client-side as truncated streams; 001.012's error UX handles that. Optionally wrap the stream with an error-event handler, but this complicates the return type.

### Low (4)

**L-001 · Model name `claude-sonnet-4-6` — verify against actual Anthropic model IDs**

- Severity: LOW
- File: `src/lib/chat/anthropic.ts` Step 4
- Details: Architecture doc says `claude-sonnet-4-6`. Anthropic's model naming has recently shifted; the actual production model ID may be `claude-sonnet-4-6-20250115` or similar. The plan uses the short form and makes it env-tunable, so Maria can override via `CHAT_MODEL` at deploy time without a redeploy.
- Suggested fix: Accept. Document in `anthropic.ts` comment that Maria should set `CHAT_MODEL` to the specific dated model if the short form resolves incorrectly.

**L-002 · No test for `X-Governed-By` header on 200 response path**

- Severity: LOW
- File: Test plan Step 9
- Details: Route tests exercise error responses (which include the header per `jsonError`). No explicit assertion for the header on the 200-streaming path. Plan includes it in the 200 response; worth a test.
- Suggested fix: Add a test case: `expect(response.headers.get('X-Governed-By')).toBe('bines.ai')` on the mocked-success path.

**L-003 · System-prompt test asserts section presence via substring — fragile if prompt text changes**

- Severity: LOW
- File: Test plan `system-prompt.test.ts`
- Details: "Required sections present" tests check for specific strings like "you are an AI trained". If Maria tweaks prompt wording, tests break.
- Suggested fix: Use looser regex or structural markers (e.g., `## Voice` as section headers).

**L-004 · `rate-limit.ts` caches module-level `_ratelimit` — edge runtime may instance-share across requests**

- Severity: LOW
- File: Step 5
- Details: Edge runtime isolates per-request by default — module-level state is safe across requests in the same isolate but may not persist. `_ratelimit` being `null`-checked per-call is correct; the optimisation (cache the instance) is safe because even if the cache is warm, re-creating it would also be fine.
- Suggested fix: None. Clean code.
