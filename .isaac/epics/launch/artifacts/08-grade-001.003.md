### Medium (2)

**M-001 · `z.iso.date()` is Zod-4-only; confirms `package.json` but not typed-lib version**

- Severity: MEDIUM
- File: `src/lib/content/types.ts` (proposed in plan Step 2)
- Details: The plan uses `z.iso.date()` — a Zod 4 API. Verified by running the installed `zod@4.3.6` locally: `z.iso.date` is a function, returns a schema that validates `YYYY-MM-DD` strings and rejects non-ISO formats like `"22/04/2026"`. However, the plan's schema tests assert this indirectly (`rejects non-ISO-date published`). Recommend adding a direct test case like `expect(() => FIELDWORK_FRONTMATTER.parse({...VALID, published: '2026-13-40'})).toThrow()` to confirm the date validator is doing the work, not just the type check.
- Impact: Low — the rejection test already covers the worst case (entirely wrong format). Calendar-date validity (month ≤ 12, day valid for month) is a nicety but not critical for the story ACs.
- Suggested fix: Add one specific calendar-invalid-date test case in `types.test.ts`. One-line addition.

**M-002 · `getFieldworkGroupedByStatus` is in scope for 001.009 per story; plan lands it here**

- Severity: MEDIUM
- File: `06-story-001.003.md` technical task list vs `07-plan-001.003.md` Step 5
- Details: Story 001.003's task list includes `getAllFieldwork`, `getFieldworkBySlug`, `getFieldworkByStatus`. Story 001.009 explicitly says *"Extend `src/lib/content/fieldwork.ts` with `getFieldworkGroupedByStatus()` helper if not already present"*. The plan adds `getFieldworkGroupedByStatus` in 001.003, pre-empting 001.009. This is fine (the "if not already present" wording authorises it), but it *does* widen this story's scope slightly.
- Impact: Minor scope bleed — however, the helper is tiny (partitions an array into 4 buckets), purely additive, does not modify ACs 001-010, and eliminates a trivial refactor in 001.009. Not a blocker.
- Suggested fix: Keep as planned — note explicitly in commit message under "Added ahead of schedule for 001.009." No plan change needed.

### Low (3)

**L-001 · Plan's test for malformed frontmatter catches only one malformation**

- Severity: LOW
- File: `src/lib/content/__tests__/fieldwork.test.ts` (proposed)
- Details: The "malformed frontmatter" test writes only `status: archived` (invalid enum) and asserts the error. Doesn't exercise: missing required field, wrong type, bad tag format. These ARE covered in `types.test.ts` as pure-schema tests, so the loader doesn't need to re-exercise them — but a one-liner checking that a completely empty frontmatter also throws would harden the error path.
- Suggested fix: Add `await fs.writeFile(path.join(tmp, 'fieldwork', 'empty.mdx'), '---\n---\nbody');` and assert both the file path and the absence-of-required-field message appear. Low-effort hardening.

**L-002 · `server-only` is added as a runtime dep; could be `devDependency`**

- Severity: LOW
- File: `package.json` (Step 1)
- Details: `server-only` is a tiny package (~1KB) whose only purpose is to throw at bundle time when imported from a client component. It's safe either way, but convention in the Next.js community is `dependencies` (not dev) because it's imported at runtime in server-only modules even in production. Plan follows convention.
- Suggested fix: Keep as `dependencies`. No change needed. Called out for awareness.

**L-003 · Plan uses `ZodType<T>` which is the base type — `ZodTypeAny` or the specific schema type is typical**

- Severity: LOW
- File: `src/lib/content/mdx.ts` (proposed Step 3)
- Details: The signature `readMdxFile<T>(filePath: string, schema: ZodType<T>)` works in Zod 4 (verified: `ZodType` is a function/class exported). Some projects prefer `ZodTypeAny` or `z.ZodSchema<T>`. Zod 4 exports `ZodType` as a top-level class so this is valid.
- Suggested fix: Keep as planned. If TypeScript complains about variance, swap to `z.ZodType<T>` or import `ZodSchema` — both equivalent at callsite.
