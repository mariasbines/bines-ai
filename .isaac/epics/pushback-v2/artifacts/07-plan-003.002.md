## Design decisions

### 1 · Runtime — switch `/api/chat` from Edge to Node

The architecture's preface lookup says "file read + Zod parse, ~5ms" and "lookup on every request" (`03-architecture.md:276`). `getFieldworkBySlug` reads MDX via `node:fs` (`src/lib/content/mdx.ts:4`); `node:fs` is **not available in Vercel Edge Runtime**. The existing `/api/chat/route.ts` declares `export const runtime = 'edge'`; calling `getFieldworkBySlug` at request time on Edge would crash at first invocation.

Two options were considered:

**A. Switch route to `runtime = 'nodejs'`.** Single-line change. All current dependencies work in Node:
- Web Crypto (`crypto.subtle.digest`, `crypto.randomUUID`) — Node 18+ exposes the same API as Edge ✓
- `@anthropic-ai/sdk` — Node-first ✓
- `@vercel/blob` — Node + Edge both ✓
- `next/server` `after()` — Node + Edge both ✓
- `node:fs` (via `getFieldworkBySlug`) — Node only ✓

**B. Build-time slug → preface map.** Generate a static JSON at build time, import as ESM. Keeps Edge runtime. Costs: a build-time script + an import side-effect coupling to MDX content. Requires the chat to redeploy whenever a piece's title/excerpt changes (acceptable — pieces ship via PR anyway, which redeploys).

**Decision: A.** Reasoning:
- Mechanical, lowest-novelty change.
- Cold-start delta (~100ms Node vs ~10ms Edge) is rounding error on a streaming LLM call (8s+ p99).
- Vercel's Node runtime supports the same `runtime = 'nodejs'` declaration; max-duration default is fine for streaming.
- B introduces a build-step coupling that the architecture doc didn't ask for; smaller surface change wins.
- The architecture's "Edge runtime ... no Node `fs`" comment (line 749) refers to the **new** routes (`run`, `sweep`); it does not apply to the existing `/api/chat` route, which the architecture line 257 explicitly amends ("MODIFY — surgical").

This is a clarifying decision, not a scope expansion. Documented in the commit message.

### 2 · Sticky `from_slug` capture — `useSearchParams` + lazy initial state

`<ChatInterface>` reads `?from` once on mount and never re-evaluates. Implementation:

```tsx
import { useSearchParams } from 'next/navigation';

export function ChatInterface() {
  const searchParams = useSearchParams();
  const [fromSlug] = useState<string | null>(() => {
    const raw = searchParams?.get('from') ?? null;
    if (raw === null || raw === '') return null;
    return raw;
  });
  // ...
}
```

Lazy initial state (`useState(() => ...)`) ensures the read happens once on mount only. Subsequent re-renders (from typing in the textarea, streaming deltas, etc.) do not re-evaluate the search-params getter. Sticky for the component instance lifetime — matches AC-001 + AC-003 (d).

Empty string `?from=` (no value after `=`) coerces to `null` — AC-003 (c).

Threaded into `postChat`:

```tsx
const result = await postChat(history, {
  signal: controller.signal,
  conversation_id,
  from_slug: fromSlug,  // was: null in story 003.001
  onDelta: ...,
});
```

Story 003.001's hard-coded `from_slug: null` becomes the captured value. The placeholder comment is removed.

### 4 · `buildPiecePreface(piece)` — locked draft, lowercase, British English

Per AC-004 + architecture-locked draft:

```ts
import type { Fieldwork } from '@/lib/content/types';

export function buildPiecePreface(piece: Fieldwork): string {
  const title = piece.frontmatter.title;
  const excerpt =
    piece.frontmatter.excerpt && piece.frontmatter.excerpt.length > 0
      ? piece.frontmatter.excerpt
      : piece.body.slice(0, 200);
  return `the visitor came here from your piece "${title}". the piece argues:
${excerpt}.
they have something to push back on. let them. don't restate the piece —
make them work for it. quote them back when their argument is sharp.`;
}
```

The fallback to `piece.body.slice(0, 200)` is defensive. The current Zod schema (`FIELDWORK_FRONTMATTER`) requires `excerpt: z.string().min(1)` — any valid Fieldwork piece has an excerpt. The fallback path is only reachable if the schema is loosened in a future story or if a piece is constructed manually in a test fixture. AC-005 (d) asserts the function "still renders without throwing" — the fallback satisfies that.

Voice properties of the locked draft:
- Lowercase opener ("the visitor"). AC-005 (b).
- British English ("argues", "make them work for it" — colloquial; no Americanisms).
- Imperative micro-instructions ("let them", "don't restate", "quote them back").
- ≤ 200 words combined with title + excerpt — preserves the SYSTEM_PROMPT budget.

The draft text matches `03-architecture.md:248-253` verbatim. No improvisation.

**Voice-check sign-off (AC-008):** the text is the architecture-locked draft, voice-checked at the architecture phase. Maria's per-story sign-off recorded as: "preface text matches architecture-locked draft (`03-architecture.md:248-253`); deferred to dev-branch review per autopilot run instructions". This is documented in the commit message; if Maria objects on dev review, follow-up story tweaks the string (cheap — single-file edit).

### 5 · Route — preface composition before Sonnet stream call

In `/api/chat/route.ts`, between the easter-egg pre-filter (step 7) and the Sonnet stream call (step 9), add:

```ts
// Phase B — story 003.002. Piece-aware preface.
let systemPrompt = SYSTEM_PROMPT;
if (from_slug !== null) {
  try {
    const piece = await getFieldworkBySlug(from_slug);
    if (piece !== null) {
      systemPrompt = buildPiecePreface(piece) + '\n\n' + SYSTEM_PROMPT;
    }
  } catch {
    // Defensive — getFieldworkBySlug is null-on-missing, but a partial
    // content-validation regression elsewhere shouldn't crash the chat.
    // Silently fall through with the baseline prompt.
  }
}
```

Then pass `system: systemPrompt` to `client.messages.stream(...)` (was `system: SYSTEM_PROMPT`).

Why try/catch despite `getFieldworkBySlug` being null-on-missing: the function calls `readMdxFile` under the hood, which throws `ContentValidationError` if **another** piece in the directory has bad frontmatter (since `getAllFieldwork` validates everything before filtering). A regression in an unrelated MDX file would otherwise crash chat. The catch swallows that risk for the preface specifically; the chat still works without piece-awareness.

### 6 · `from_slug` lifecycle on the route

- `from_slug` from validated body → stored as a const (already done in 003.001).
- Used as a Map-style lookup key into `getFieldworkBySlug` → never interpolated into the prompt as a raw string. AC-008.
- `null`, empty string, unknown slug, path-traversal-shaped → all converge on "no preface". AC-006 + AC-007 + AC-009 (a, c, d, e).
- The log entry **always** carries `from_slug` verbatim — even when the slug was unknown. This is AC-009 (f): preserves attribution for the (small) admin-debug case where Maria spots a bad CTA in the field. The loader filter at build-time will drop unknown-slug verdicts — that's downstream.

### 7 · Empty-string treatment

`?from=` with no value after the `=` produces `searchParams.get('from')` → `''` (empty string). Three places convert to null:

1. **Component side** (lazy state init): `if (raw === null || raw === '') return null;` — AC-003 (c).
2. **Wire side**: `from_slug: ''` on the body → zod accepts (`.string().nullable().optional()`), validates as `''`. Route then `from_slug ?? null` keeps the empty string. **Fix**: route normalises `from_slug = body.from_slug || null` (truthy fallback) so the empty string maps to null on the server too. AC-009 (e).

Choosing component-side normalisation as the canonical place; route-side is defence-in-depth.

Update to step 11 in 003.001's plan: change `from_slug = v.from_slug ?? null` to `from_slug = v.from_slug || null` so empty string → null. Backwards-compat: 003.001's tests didn't supply empty string for from_slug; nothing regresses.
