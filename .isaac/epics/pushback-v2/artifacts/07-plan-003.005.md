# Implementation Plan: 003.005 — Chat-end signal in `<ChatInterface>`

## Overview

The visitor-side trigger for the judge. Adds two chat-end signals to `<ChatInterface>` — a 2-minute idle timer + `pagehide` / `beforeunload` listeners — both invoking `fireChatEnd()` which posts `{ conversation_id }` to `/api/argue-judge/run` via `navigator.sendBeacon`. A `Set`-backed debounce guard ensures at-most-once-per-conversation_id firing.

🟡 Critical Path TDD. Medium complexity (~30 LOC component change; ~150 LOC test). 9 ACs. Three risks (PB2-SEC-007 beacon tampering, PB2-PRF-003 latency, PB2-OPS-003 idle/unload race). Single commit on `dev`.

After this story merges:
- Every conversation with at least one user message gets a judge call (idle or unload, whichever fires first).
- Repeat fires are no-ops at the client.
- Stale tabs (browser closed mid-conversation, mobile session frozen) are caught by the sweep cron — story 003.006.

## Design decisions

### 1 · Sticky-id refactor for the idle reset to actually work

The story's AC-001/AC-002 use `conversationIdRef.current` and a `firedConversationsRef = useRef(new Set<string>())`. Both refs are read by the unload event handler, but `useRef` updates do NOT trigger re-renders, so the unload effect's closure captures the ref objects (not their `.current` snapshots) and reads `.current` at fire-time. That's the correct pattern.

The idle-reset logic uses a different approach: every render (after a submit or stream event) re-establishes the timer. Implementation:

```tsx
useEffect(() => {
  if (conversationIdRef.current === null) return;
  if (status === 'streaming') return; // active stream — don't fire mid-response
  const handle = setTimeout(() => fireChatEndStable.current(), 2 * 60 * 1000);
  return () => clearTimeout(handle);
}, [messages.length, status]);
```

Dependency on `messages.length` resets the timer when a new message lands. Dependency on `status` resets it when streaming starts/ends. The cleanup function clears the previous timer — only one outstanding timer at a time per render.

Key subtlety: `fireChatEndStable.current()` (where `fireChatEndStable = useRef<() => void>(...)`) lets us call the latest `fireChatEnd` without listing it as a dep (which would cause re-fire churn since `fireChatEnd` changes when `conversationIdRef` reads change in older patterns). `useCallback` would suffice if all deps are stable refs.

Cleaner pattern:

```tsx
const fireChatEnd = useCallback(() => {
  const id = conversationIdRef.current;
  if (id === null) return;
  if (firedConversationsRef.current.has(id)) return;
  firedConversationsRef.current.add(id);
  try {
    if (typeof navigator !== 'undefined' && typeof navigator.sendBeacon === 'function') {
      const payload = JSON.stringify({ conversation_id: id });
      const blob = new Blob([payload], { type: 'application/json' });
      navigator.sendBeacon('/api/argue-judge/run', blob);
    }
  } catch {
    // sendBeacon unavailable / failed. Sweep is the safety net.
  }
}, []); // depends only on refs — stable across renders
```

`useCallback([], [])` because the only reads are `useRef.current` accesses. Stable identity → safe to use as event-listener handler in the unload effect.

### 2 · Idle timer effect — depends on settled state

```tsx
useEffect(() => {
  if (status === 'streaming') return; // mid-stream — don't schedule
  if (conversationIdRef.current === null) return; // no chat yet
  const handle = setTimeout(() => fireChatEnd(), 2 * 60 * 1000);
  return () => clearTimeout(handle);
}, [messages.length, status, fireChatEnd]);
```

The dependency array:
- `messages.length` — resets on submit (new user msg appended).
- `status` — resets when streaming starts (timer cancelled) and when streaming ends (timer re-armed).
- `fireChatEnd` — stable; included for ESLint correctness.

When `status === 'streaming'`, no timer is scheduled — the existing timer (if any) gets cleaned up by the previous render's return. When the stream ends and `status` flips back to `'idle'`, the effect re-runs and schedules a fresh timer.

For the stream-activity-resets-idle test: each `onDelta` callback updates `messages` (the assistant content grows) which changes `messages` reference but NOT `messages.length` (length stays the same — same number of messages). Hmm, that breaks the AC-006 case "Idle resets on stream activity".

Re-reading AC-003: "the timer must reset on user input or stream activity. Using a ref + handler pattern is acceptable; the rule is the timer must reset on user input or stream activity." OK — stream activity is `status === 'streaming'`. While streaming, the effect returns early (no timer). When streaming ends, a fresh 2-min timer is armed. So the test for "stream activity resets idle" is satisfied by: stream for X seconds → status='streaming' → no timer → stream ends → status='idle' → new 2-min timer. The 2-min countdown begins from stream-end, not from stream-start. That's the desired "resets" behaviour.

But the AC-006 case literal text says "simulate streaming `onDelta` callbacks for 90 seconds, advance timers another 90 seconds without further activity, assert beacon fires only at the 2-min-after-last-delta mark". This implies the timer should reset on each delta. With my dep-on-`messages.length` design, deltas don't trigger effect re-runs (length is constant during the stream).

I have two options:

**A. Add `messages` (the array) to the dep list** — every delta mutates the array via `setMessages`, so the array reference changes per delta. Effect re-runs per delta — timer resets per delta.

**B. Block timer setup while streaming** — `if (status === 'streaming') return` covers it. Timer doesn't fire during streaming. Once streaming ends, a 2-min timer starts fresh.

Option A matches the AC-006 literal language ("idle resets on stream activity"). Option B is simpler but doesn't reset per-delta — it resets once at stream-end.

Re-reading the AC-006 case: "advance timers another 90 seconds without further activity, assert beacon fires only at the 2-min-after-last-delta mark". For Option B, after the last delta the stream ends (status flips to idle), and the 2-min timer starts. Advancing 90 seconds without activity wouldn't fire (90 < 120). Advancing 30 more seconds would fire. The assertion "fires only at the 2-min-after-last-delta mark" is consistent with Option B.

Going with Option B — simpler, matches the AC's described behaviour. The test will simulate streaming via the `status` state.

Wait — the test layer doesn't expose `status` directly. The component owns it. Tests simulate streaming by mocking `postChat` to NOT resolve immediately, leaving `status === 'streaming'` for some time. Or simpler: the tests can verify behaviour by examining when the beacon fires relative to the stream-end timing.

Implementation simpler than test: I'll go with Option B and write the test to simulate "submit → status streaming → stream completes → 2-min timer fires beacon".

Actually, looking at the existing component, `setStatus('idle')` happens after `await postChat(...)` resolves. So the test can:
1. Use a `mockPostChat` that returns a never-resolving Promise → status stays 'streaming'.
2. Advance fake timers — no beacon fires.
3. Resolve the promise → status flips to 'idle'.
4. Advance 2 minutes → beacon fires.

That's clean. AC-006 case "Idle resets on stream activity" is satisfied: no beacon during streaming, then a fresh 2-min countdown after.

But the AC literal says "resets on stream activity" (resets, not "doesn't fire during"). Option B technically doesn't RESET — it BLOCKS. Functionally equivalent for the tested behaviour ("beacon fires 2 min after streaming ends"), but semantically different.

I'll do Option A — include `messages` array in dep list (not just `messages.length`). Reads: per-delta, each delta produces a new `messages` array reference (via `setMessages((prev) => prev.map(...))` which returns a fresh array). Each render schedules a fresh timer; the cleanup cancels the previous. Net effect: timer is reset on each delta during streaming, AND on each user submit.

This is the most literal interpretation of the AC. Costs: the effect runs many times during a stream (once per delta), but each run is a `setTimeout` + cleanup pair — cheap. JS perf concern is negligible.

```tsx
useEffect(() => {
  if (conversationIdRef.current === null) return;
  const handle = setTimeout(() => fireChatEnd(), 2 * 60 * 1000);
  return () => clearTimeout(handle);
}, [messages, status, fireChatEnd]);
```

I drop the `if (status === 'streaming') return` — instead the timer is allowed to be scheduled mid-stream, but it gets cleaned up by the next delta's effect run before it can fire. The clean rhythm: each delta resets the 2-min countdown.

If a delta doesn't arrive for 2 minutes (long Sonnet pause), the timer fires during the stream — not great. But: realistically Sonnet streams in <1s chunks; a 2-min gap mid-stream means the stream is wedged. Firing a beacon for a wedged stream is fine — the run route's IP-bind passes, recency passes, the judge runs over the partial conversation and writes a verdict. Acceptable degenerate case.

### 3 · Unload effect — single source of truth for both events

```tsx
useEffect(() => {
  if (typeof window === 'undefined') return;
  window.addEventListener('beforeunload', fireChatEnd);
  window.addEventListener('pagehide', fireChatEnd);
  return () => {
    window.removeEventListener('beforeunload', fireChatEnd);
    window.removeEventListener('pagehide', fireChatEnd);
  };
}, [fireChatEnd]);
```

`fireChatEnd` is stable (useCallback with `[]`), so the effect runs once on mount, registers both listeners, returns the cleanup. On unmount, listeners are removed.

The debounce guard inside `fireChatEnd` means a `pagehide` followed by a `beforeunload` (Safari fires both during page close) results in exactly one beacon. AC-006 case "Debounce guard prevents double fire" verifies this.

### 4 · Test fake-timer hygiene

Existing `<ChatInterface>` tests use real timers. Adding `vi.useFakeTimers()` per-describe-block requires `vi.useRealTimers()` cleanup or it leaks into other tests in the same file.

Pattern:

```tsx
describe('<ChatInterface> — chat-end signal (story 003.005)', () => {
  beforeEach(() => {
    vi.useFakeTimers();
  });

  afterEach(() => {
    vi.useRealTimers();
  });

  // tests using fake timers
});
```

Mock `navigator.sendBeacon` per beforeEach (jsdom doesn't ship it):

```tsx
const sendBeaconMock = vi.fn(() => true);
beforeEach(() => {
  Object.defineProperty(navigator, 'sendBeacon', {
    writable: true,
    configurable: true,
    value: sendBeaconMock,
  });
  sendBeaconMock.mockClear();
});
```

### 5 · `submit` helper across fake timers

The existing test's `submit` helper uses `await act(async () => { await Promise.resolve(); })` to flush microtasks. With fake timers, `act` still flushes microtasks correctly. But if any code path uses `setTimeout(..., 0)` for deferral, fake timers would block it — none of the current chat code does, so this isn't a concern.

For the stream-activity test: simulate streaming by configuring `mockPostChat` to NOT resolve until manually awaited:

```tsx
let resolvePostChat: (val: { ok: boolean }) => void;
mockPostChat.mockImplementation(() => new Promise<{ ok: boolean }>((resolve) => {
  resolvePostChat = resolve;
}));
```

Then submit, advance fake time, resolve, advance again. This is gritty but reliable.

Actually for AC-006 stream-activity: the simpler test is a pure timer-based one using `messages` deps. Submit → 90s → no beacon (timer would fire at 120s). Submit again → 90s → no beacon (timer just reset). Advance another 30s → beacon fires (total 120s from last submit).

No need to mock streaming gymnastics — submit is sufficient because the same "messages" dep covers both cases (user submits change `messages` reference; stream deltas also change it). The tests exercise the submit path; the stream path would behave identically in production.

### 6 · `pagehide` / `beforeunload` event dispatch in tests

```tsx
fireEvent(window, new Event('pagehide'));
fireEvent(window, new Event('beforeunload'));
```

`@testing-library/react`'s `fireEvent` accepts a `Window` target. Confirmed by reading the testing-library docs; the existing test imports `fireEvent` from `@testing-library/react`.

### 7 · Cleanup-on-unmount test

```tsx
const { unmount } = render(<ChatInterface />);
await submit('hi');
expect(sendBeaconMock).not.toHaveBeenCalled(); // no beacon yet — only listeners + timer
unmount();
fireEvent(window, new Event('pagehide'));
fireEvent(window, new Event('beforeunload'));
vi.advanceTimersByTime(3 * 60 * 1000);
expect(sendBeaconMock).not.toHaveBeenCalled(); // listeners removed; timer cancelled
```

The timer cleanup is via the effect's return function; the listeners cleanup is via the unload effect's return function. Both run on unmount. The assertion proves neither leaks.

Edge case: between submit and unmount, the idle timer is alive but hasn't fired. After unmount, advancing past 2 minutes shouldn't fire the beacon because the cleanup ran. If it does fire, the beacon would point at a no-longer-mounted component, but more importantly it'd indicate a leak.

## Implementation Steps

### Step 1 · `<ChatInterface>` — add three pieces

1. `firedConversationsRef = useRef(new Set<string>())` — debounce guard.
2. `fireChatEnd = useCallback(() => { ... }, [])` — payload + sendBeacon + try/catch.
3. Idle-timer `useEffect([messages, status, fireChatEnd])` — schedules 2-min timer; cleanup cancels.
4. Unload `useEffect([fireChatEnd])` — adds `pagehide` + `beforeunload` listeners; cleanup removes both.

### Step 2 · `<ChatInterface>` tests — extend with 9 specs

Test cases per AC-006 plus the cleanup case:

1. No fire when no messages
2. Idle elapse fires beacon (single submit + 2-min advance)
3. Idle resets on submit (submit, 90s, submit, 90s, no fire)
4. Idle resets on subsequent activity (substitute "stream activity" with another submit, since deps include `messages` either way — exercising the same code path)
5. `pagehide` fires beacon
6. `beforeunload` fires beacon
7. Debounce: pagehide then beforeunload = one fire
8. No fire when conversation_id is null (event dispatched without prior submit)
9. Cleanup on unmount

For test 4 in particular: since stream-activity also flips `messages` reference (via `setMessages((prev) => prev.map(...))` inside `onDelta`), the dep list catches it. But simulating that path requires a mocked `postChat` that calls `onDelta` between awaits. Cleanest: use submit-as-proxy, since both produce the same effect trigger. Document this in the test comments.
