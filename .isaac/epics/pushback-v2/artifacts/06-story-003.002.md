# Story 003.002: ?from=<slug> capture + chat-route preface (Phase B)

**Epic:** `pushback-v2`
**Phase:** B — judge populates silently (no public render yet)

## Description

The piece-aware-chat half of Phase B's UX wiring. `<ChatInterface>` reads `?from=<slug>` from the URL once on mount (via `useSearchParams`), captures it sticky for the component lifetime, and threads it into every `/api/chat` POST body. The chat route then looks up the piece via `getFieldworkBySlug(from_slug)` and prepends a soft preface to the system prompt — anchoring the conversation to the piece the visitor came from. If the slug is unknown or malformed, the preface is skipped silently and the chat behaves as if `from_slug` were absent (defence against attacker-controlled slug injection).

This story does **not** ship the chat-end beacon (003.005), the judge module (003.003), or any public render (003.007). After this story merges, `argue-log` entries from `/argue?from=<slug>` carry the slug, and Sonnet's system prompt is piece-aware — but no `argue-judges/` writes happen yet (the judge route doesn't exist in this story).

## Acceptance Criteria

- [ ] AC-001: `<ChatInterface>` reads `?from=<slug>` via `useSearchParams().get('from')` once on first render, stores in `useState<string | null>(null)` (sticky for component lifetime — never re-evaluated on re-render), and passes the captured value into every `postChat({ from_slug })` call. Mid-conversation pivots (visitor opens a new tab and pastes a different `?from=`) do **not** re-tag — Q3 sticky-at-start.
- [ ] AC-002: When the URL has no `?from=` query param, `<ChatInterface>` passes `from_slug: null` to `postChat`. Identical behaviour to story 003.001.
- [ ] AC-003: `src/components/__tests__/ChatInterface.test.tsx` extended with: (a) URL contains `?from=valid-slug` → `postChat` body has `from_slug: 'valid-slug'`; (b) URL has no `from` param → `postChat` body has `from_slug: null`; (c) URL contains `?from=` (empty value) → treated as null; (d) sticky behaviour: simulating a route change (or a re-render with different searchParams) does NOT change the captured `from_slug` after first render.
- [ ] AC-004: `src/lib/chat/system-prompt.ts` exports a new function `buildPiecePreface(piece: Fieldwork): string` that returns the locked-draft preface text:

  ```
  the visitor came here from your piece "{title}". the piece argues:
  {excerpt}.
  they have something to push back on. let them. don't restate the piece —
  make them work for it. quote them back when their argument is sharp.
  ```

  Where `{title}` is `piece.frontmatter.title` and `{excerpt}` is the piece's frontmatter `excerpt` (or first ~200 chars of the body if frontmatter has no `excerpt` field — defensive fallback). Lowercase, British English, no exclamation. Voice-checked draft per architecture §`system-prompt.ts`. **Maria sign-off on this string before AC-008 closes** — recorded in commit message or story comment.
- [ ] AC-005: `src/lib/chat/__tests__/system-prompt.test.ts` extended with: (a) `buildPiecePreface(piece)` returns a string containing both the title and the excerpt; (b) preface starts with lowercase opener; (c) preface contains the canonical instruction phrase "let them. don't restate the piece"; (d) given a piece with no excerpt, the preface still renders without throwing (uses body fallback).
- [ ] AC-006: `src/app/api/chat/route.ts` modified: when the validated request body has `from_slug !== null`, the route calls `getFieldworkBySlug(from_slug)` server-side. If the lookup returns a piece, prepend `buildPiecePreface(piece)` to the existing `SYSTEM_PROMPT` and pass the combined string to `messages.stream`. If the lookup returns `null` (unknown slug, path-traversal-shaped input, malformed string), the preface is **skipped silently** — no error, no log spam, chat behaves as if `from_slug` were null.
- [ ] AC-007: The lookup never reaches a filesystem path-resolution. `getFieldworkBySlug` performs an exact-match against parsed frontmatter `slug` fields — `from_slug = '../../etc/passwd'` (or any other path-traversal-shaped string) returns `null` and the preface is skipped. Verified by route test fixture (AC-009 case (d)).
- [ ] AC-008: `from_slug` from the URL is **never** inserted into the system prompt as raw user input. The preface only contains `piece.frontmatter.title` and `piece.frontmatter.excerpt` — both author-controlled values from the MDX file. The visitor-controlled slug parameter is used only as a Map-style lookup key, never as a string interpolation source in the prompt.
- [ ] AC-009: `src/app/api/chat/__tests__/route.test.ts` extended with: (a) `from_slug: null` → no preface, system prompt unchanged from baseline; (b) `from_slug: 'valid-slug'` (mock returns a piece) → system prompt sent to Anthropic mock contains the preface text; (c) `from_slug: 'unknown-slug'` (mock returns null) → no preface, no error, chat proceeds normally; (d) `from_slug: '../../etc/passwd'` (mock returns null) → no preface, no error, no filesystem access; (e) `from_slug: ''` (empty string) → treated as null, no preface; (f) log entry written by this request carries `from_slug` verbatim (the value the client sent — preserves attribution even when the slug is invalid; the loader filter handles it later).
- [ ] AC-010: Quality gates green: `pnpm typecheck && pnpm lint && pnpm test` all pass.
