# Story 003.005: Chat-end signal in <ChatInterface> (Phase B)

**Epic:** `pushback-v2`
**Phase:** B — judge populates silently (no public render yet)

## Description

The visitor-side trigger for the judge. `<ChatInterface>` adds two chat-end signals — (a) a 2-minute idle timer that resets on every user submit and every streaming `onDelta` callback, and (b) `pagehide` + `beforeunload` listeners — both of which call `fireChatEnd()` to send a `{ conversation_id }` payload via `navigator.sendBeacon` to `/api/argue-judge/run`. A debounce guard (`useRef<Set<string>>`) ensures `sendBeacon` fires at most once per `conversation_id` for the lifetime of the component. `sendBeacon` is the only synchronous network primitive that survives unload reliably across browsers; `fetch({ keepalive: true })` works on Chrome/Firefox desktop but not Safari iOS.

After this story merges, every visitor conversation that gets at least one user message gets a judge call (idle or unload, whichever fires first). The `/api/argue-judge/run` route from 003.004 handles the rest. Sweep cron from 003.006 covers any conversation where neither signal fires (browser closed during a long pause, mobile session frozen, etc.).

## Acceptance Criteria

- [ ] AC-001: `<ChatInterface>` adds a `useRef<Set<string>>` (call it `firedConversationsRef`) initialised to a fresh `Set`. Used as the debounce guard so `fireChatEnd` for a given `conversation_id` runs exactly once for the component lifetime.
- [ ] AC-002: `fireChatEnd()` is a stable callback (e.g. `useCallback` referencing the conv-id ref + the firedConversationsRef): if `conversationIdRef.current === null` → return early (no messages sent yet, nothing to judge); if `firedConversationsRef.current.has(conversationIdRef.current)` → return early (already fired); otherwise: add the id to the set, build payload `JSON.stringify({ conversation_id: conversationIdRef.current })`, wrap in `new Blob([payload], { type: 'application/json' })`, call `navigator.sendBeacon('/api/argue-judge/run', blob)`. The function never `await`s anything (sendBeacon is synchronous fire-and-forget).
- [ ] AC-003: **Idle timer effect**. A `useEffect` registers a `setTimeout(fireChatEnd, 2 * 60 * 1000)` whenever `conversationIdRef.current !== null` AND messages have settled (no in-flight stream). The dependency array includes `messages.length` and `status` (the existing chat status state) so the timer resets on every user submit and every assistant streaming completion. Cleanup function cancels the previous timeout — guarantees one pending timer at a time. Using a ref + handler pattern is acceptable; the rule is the timer must reset on user input or stream activity.
- [ ] AC-004: **Unload effect**. A separate `useEffect` adds `window.addEventListener('beforeunload', fireChatEnd)` AND `window.addEventListener('pagehide', fireChatEnd)`. Cleanup removes both. Both events trigger the same `fireChatEnd` (single source of truth). `pagehide` is the mobile-reliable variant; `beforeunload` covers desktop including refresh + close; firing both is safe because the debounce guard (AC-002) makes the second a no-op.
- [ ] AC-005: `<ChatInterface>` does NOT subscribe to any other chat-end-shaped event. No visibility-change listener (over-fires when user switches tabs); no inactivity-with-mouse-idle (over-engineered); just the two architectural signals. The simpler the surface the easier to debug.
- [ ] AC-006: `src/components/__tests__/ChatInterface.test.tsx` extended with the following cases (vitest + @testing-library/react + jsdom):
  - **No fire when no messages sent**: render component, advance fake timers by 5 minutes (more than the 2-min idle), assert `navigator.sendBeacon` was NOT called (conversationIdRef is still null).
  - **Idle elapse fires beacon**: render, simulate one user submit (mints conversation_id, captures it), advance fake timers by 2 minutes, assert `navigator.sendBeacon` called once with the matching `conversation_id` payload.
  - **Idle resets on submit**: render, submit, advance 90 seconds, submit again, advance 90 seconds, assert `navigator.sendBeacon` NOT called yet (timer reset on second submit).
  - **Idle resets on stream activity**: render, submit, simulate streaming `onDelta` callbacks for 90 seconds, advance timers another 90 seconds without further activity, assert beacon fires only at the 2-min-after-last-delta mark.
  - **`pagehide` fires beacon**: render, submit (mints id), dispatch `window` `pagehide` event, assert beacon called with the matching payload.
  - **`beforeunload` fires beacon**: render, submit, dispatch `beforeunload`, assert beacon called.
  - **Debounce guard prevents double fire**: render, submit, fire `pagehide`, then dispatch `beforeunload`, assert `sendBeacon` called exactly **once** (set membership prevented the second).
  - **No fire when conversation_id is null**: render component, dispatch `pagehide` without any prior submit, assert `sendBeacon` NOT called.
  - **Cleanup on unmount**: render, submit, unmount component, assert no `pagehide`/`beforeunload` listeners remain on `window` (memory leak guard).
- [ ] AC-007: The beacon payload shape matches what `/api/argue-judge/run` validates: `{ conversation_id: <uuid string> }`. Content-Type `application/json` (set via Blob type). No additional fields — keep the surface minimal.
- [ ] AC-008: `navigator.sendBeacon` is called inside a `try/catch` no-op safety net for environments where the API is missing (very old browsers, jsdom edge cases). On unavailable `sendBeacon`, log nothing (no console spam) and let the sweep cron be the safety net.
- [ ] AC-009: Quality gates green: `pnpm typecheck && pnpm lint && pnpm test`.

## Scope rules

- Do NOT add a `visibilitychange` listener. Only `beforeunload` + `pagehide` per architecture.
- Do NOT call `fetch` or any non-sendBeacon network primitive for chat-end. sendBeacon is the only one that survives unload reliably.
- Do NOT change the idle interval from 2 minutes. Locked from concept Q2.
- Do NOT add error UI when sendBeacon fails. Sweep is the safety net.
- Do NOT modify the chat-route, the judge module, or the run route in this story. They were shipped in 003.001-003.004.
- Do NOT touch the public surfaces (`<PushbackSummary>` etc.). Story 003.007.
- Do NOT delete v1 surface. Story 003.009.
- Do NOT add a "send a final summary message" via `fetch` keepalive — sendBeacon-only is the architectural choice for cross-browser reliability.

## Risk mitigations (from 04-assessment.md)

- **PB2-SEC-007** beacon payload tampering → AC-007 (minimal payload — only `conversation_id`); the IP-bind in /api/argue-judge/run (003.004 AC-005) is the gate
- **PB2-PRF-003** chat-end signal latency on visitor close → AC-002 (sendBeacon is fire-and-forget, no client-side wait), AC-008 (graceful no-op on missing API)
- **PB2-OPS-003** race between idle and pagehide beacons → AC-002 + AC-006 case (debounce guard fires once per conversation_id even when both events fire)
