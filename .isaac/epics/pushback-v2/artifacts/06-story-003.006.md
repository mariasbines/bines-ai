# Story 003.006: /api/argue-judge/sweep cron + vercel.json (Phase B)

**Epic:** `pushback-v2`
**Phase:** B — judge populates silently (no public render yet)

## Description

The resilience net for the judge. A daily Vercel cron route at `/api/argue-judge/sweep` (GET, edge runtime, `CRON_SECRET` bearer auth) reads yesterday's argue-log entries, groups them by `conversation_id`, skips any that already have a verdict in `argue-judges/`, runs `judgeConversation` for the rest, and writes verdicts. Per-conversation errors are logged + counted but do NOT abort the sweep — a flaky Sonnet API blip cannot kill the whole nightly job. Schedule lands at `30 4 * * *` UTC, one hour after the existing `argue-log/cleanup` cron.

This story also adds the `maxDuration: 300` setting on the sweep path so sequential Sonnet calls (estimated ~30s wall time at v1 volume) can complete inside the function timeout. Story is the **last Phase B story** before the soft-launch gap; after merge the system has full judge coverage with no public surface.

Missing or wrong secret → 401. Missing `CRON_SECRET` env var → 500 + `[argue-judge] cron-secret-unconfigured` log.
- [ ] AC-002: Target day defaults to **yesterday UTC** (computed as `new Date(Date.now() - 24*3600*1000)` formatted via `dayKeyUtc`). Optional `?day=YYYY-MM-DD` query param overrides the target — validated against the existing `isValidDayKey` regex from `src/lib/argue-log/day.ts`. Non-matching `?day=` → 400 + `{ category: 'validation' }`. The override exists for manual replay (Maria runs `curl ... ?day=2026-04-20` to re-sweep a specific day after a fix).
- [ ] AC-003: Selection logic per architecture §sweep:
  - Read `argue-log/<targetDay>.jsonl` via `readArgueLogDay(targetDay)`.
  - Group entries by `conversation_id`. **Skip entries with `conversation_id` undefined or empty** (pre-Phase-A entries — they age out via 90-day cleanup; PB2-DAT-001 documents this cutoff).
  - Read `argue-judges/<targetDay>.jsonl` AND `argue-judges/<today>.jsonl` (verdicts may straddle the UTC boundary). Build a `Set<conversation_id>` of already-judged ids.
  - For each conversation_id NOT already judged: sort its argue-log entries by `timestamp` asc → turns array → call `judgeConversation(id, latestEntry.from_slug ?? null, turns)` → on success `appendArgueJudge(verdict)`; on throw, log `[argue-judge] sweep-failed conv_id=<id> reason=<msg>` and continue.
  - Errors do NOT abort the sweep. Per-conversation isolation is the rule.
- [ ] AC-004: Response envelope: 200 OK + `{ day: '<targetDay>', judged: N, skipped_already_judged: M, errors: K, ... }`. The `errors` count is incremented once per `judgeConversation` failure. Response carries `X-Governed-By: bines.ai` header on success and on every error path.

## Scope rules

- Do NOT migrate the sweep to concurrent Sonnet calls (`Promise.allSettled` with concurrency limit). Architecture flagged this as a deferred optimisation; trigger is conversations/day > ~30. v1 stays sequential.
- Do NOT add cron alerting / notification on errors. Manual monthly review is the v1 stance per architecture.
- Do NOT use a fresh `CRON_SECRET` for this route. Reuses the existing one (PB2-SEC-005 accepted risk — operational simplicity outweighs separate-secrets benefit).
- Do NOT delete or modify any `argue-log/*` files from this route. Sweep reads only; cleanup deletes; never crossed.
- Do NOT add the chat-end signal in `<ChatInterface>`. Story 003.005.
- Do NOT modify the public surfaces (`<PushbackSummary>` etc.). Story 003.007.
- Do NOT delete v1 surface. Story 003.009.
- Do NOT log visitor message content via `console.*`. Logs include conversation_id + error reason only.
