# Implementation Plan: 003.001 — Schema diff + conversation_id threading

## Overview

Phase A leaf story for the `pushback-v2` epic. Threads a stable `conversation_id` (UUID v4) through every chat round-trip and adds the additive `from_slug` field to the schema (left at `null` in this story — Phase B story 003.002 wires actual `?from=` capture). No judge module, no UI change, no new public surface — purely the plumbing every Phase B and Phase C story consumes.

🔴 Full TDD. Medium complexity. 11 ACs, 3 risks mapped (PB2-SEC-008, PB2-DAT-001, PB2-PRV-003). Single commit on `dev`.

The change is additive at the schema level (`schema_version` literal stays at `1`; pre-Phase-A entries continue to parse), at the wire level (legacy clients that omit `conversation_id` are tolerated — server mints a fallback UUID), and at the visitor-experience level (no observable change). Safe to ship behind no flag.

## Design decisions

### 1 · Schema additivity — both fields optional, `from_slug` nullable

`ARGUE_LOG_ENTRY` gains:

- `conversation_id: z.string().uuid().optional()` — UUID v4 by convention but the schema layer accepts any RFC 4122 UUID (zod's `.uuid()` is shape-agnostic to version). The mint side (`newConversationId`) commits to v4; the parse side stays permissive so a future v7 swap doesn't break replay.
- `from_slug: z.string().nullable().optional()` — three legal states: omitted (legacy / Phase A), `null` (Phase A places this; "no piece origin" in Phase B+), or a non-empty slug string (Phase B+ when `?from=` was present).

`schema_version` stays at the literal `1`. Phase 6 architecture explicitly carves Phase A as "additive-only schema bump without version drift". Round-trip parses on **both** old and new shapes is the canonical regression test (AC-001 / AC-002).

### 2 · `newConversationId()` — pure module, named export, no side effects

```ts
// src/lib/conversation/id.ts
export function newConversationId(): string {
  return crypto.randomUUID();
}
```

No env reads. No `Math.random` fallback (Edge + modern browsers + Node 18+ all expose `crypto.randomUUID`; jsdom 22+ exposes it via the `node:crypto` shim — already exercised by existing `<ChatInterface>` tests). Named export (not default) so `vi.mock('@/lib/conversation/id', () => ({ newConversationId: vi.fn(...) }))` is straightforward in component tests.

Rationale for putting this in `src/lib/conversation/` (not `src/lib/argue-log/` or `src/lib/chat/`): the architecture (§Components) carves out `src/lib/conversation/` for both this id helper AND the future chat-end signal (story 003.005). Pre-creating the directory in 003.001 keeps the Phase B story clean.

### 3 · Client-side mint timing — lazy on first submit, NOT on mount

Per architecture decision 1 (§Decisions resolved from Phase 2 findings) and AC-007: empty conversations don't need ids. `useRef<string | null>(null)`; mint on first `handleSubmit` only:

```tsx
if (conversationIdRef.current === null) {
  conversationIdRef.current = newConversationId();
}
const conversation_id = conversationIdRef.current;
```

ID stability across turns: same `useRef` is read on every subsequent `handleSubmit`. Resets only on hard reload (component unmount destroys the ref). No reset on submit, no reset on error, no reset on abort.

Why `useRef` (not `useState`): the id is a render-irrelevant tracking value; mutating it should not trigger a re-render. `useRef` matches `abortRef` already in the component.

### 4 · Server-side fallback mint — defence against legacy clients

Per AC-009: if a request body omits `conversation_id`, the route mints one server-side. This protects against a stale tab serving a pre-deploy `<ChatInterface>` bundle that posts without the field — the log entry still gets a UUID, just a server-minted one.

Implementation: between the validate step and the `scheduleLog()` call, compute `const conversation_id = body.conversation_id ?? newConversationId()`. The route uses `validateRequest`'s validated body for `messages`, so we pass through the validated `conversation_id` if present (zod accepts the optional field) and fall back if not. `from_slug` follows the same `?? null` pattern but stays `null` placeholder until 003.002.

Note: `validateRequest` returns a narrowed `ValidateResult` shape that today only carries `messages`. The cleanest threading change is to extend `ValidateResult.ok=true` to also carry `conversation_id?: string` and `from_slug?: string | null`, populated from `parsed.data` when present. The route then reads them off the validated result and falls back to mint where needed. This keeps zod parse errors centralised (the route never sees raw `body` after validate).

### 5 · `postChat` body shape — additive optional fields, gracefully omitted

`ChatClientOptions` gains:

```ts
export interface ChatClientOptions {
  signal?: AbortSignal;
  onDelta: (text: string) => void;
  conversation_id?: string;
  from_slug?: string | null;
}
```

Body construction:

```ts
const payload: Record<string, unknown> = { messages };
if (options.conversation_id !== undefined) payload.conversation_id = options.conversation_id;
if (options.from_slug !== undefined) payload.from_slug = options.from_slug;
JSON.stringify(payload)
```

Why `if undefined` rather than always-include: a legacy caller (no options provided) gets exactly the pre-Phase-A wire shape. Backwards compatibility at the test level is preserved (existing `client.test.ts` specs pass without modification because they never pass these new options).

Note `from_slug: null` IS included (null is meaningful — "explicit no origin"). Only `undefined` is filtered. This matches zod's three-state semantics from §Design 1.

### 6 · Route-side validate extension

`CHAT_REQUEST` schema gets:


`validateRequest` return type:



### Step 5 · Validate schema — `src/lib/chat/validate.ts`

Extend the schema and the result discriminant:

```ts
export const CHAT_REQUEST = z.object({
  messages: z
    .array(
      z.object({
        role: z.enum(['user', 'assistant']),
        content: z.string().min(1),
      }),
    )
    .min(1),
  conversation_id: z.string().uuid().optional(),
  from_slug: z.string().nullable().optional(),
});

export type ChatMessage = z.infer<typeof CHAT_REQUEST>['messages'][number];

export type ValidateResult =
  | {
      ok: true;
      messages: ChatMessage[];
      conversation_id?: string;
      from_slug?: string | null;
    }
  | { ok: false; status: 400 | 413; error: string; category: 'input' };

export function validateRequest(body: unknown): ValidateResult {
  const parsed = CHAT_REQUEST.safeParse(body);
  if (!parsed.success) {
    return { ok: false, status: 400, error: 'Invalid request shape', category: 'input' };
  }
  for (const m of parsed.data.messages) {
    if (m.content.length > MAX_MESSAGE_CHARS) {
      return {
        ok: false,
        status: 413,
        error: `Message too long (max ${MAX_MESSAGE_CHARS} chars)`,
        category: 'input',
      };
    }
  }
  const truncated = parsed.data.messages.slice(-MAX_TURNS);
  return {
    ok: true,
    messages: truncated,
    conversation_id: parsed.data.conversation_id,
    from_slug: parsed.data.from_slug,
  };
}
```

### Step 7 · Chat client — `src/lib/chat/client.ts`

```ts
import type { ChatErrorCode } from './error-messages';

export interface ChatClientOptions {
  signal?: AbortSignal;
  onDelta: (text: string) => void;
  conversation_id?: string;
  from_slug?: string | null;
}

export interface ChatClientResult {
  ok: boolean;
  errorCode?: ChatErrorCode;
}

export async function postChat(
  messages: Array<{ role: 'user' | 'assistant'; content: string }>,
  options: ChatClientOptions,
): Promise<ChatClientResult> {
  const payload: Record<string, unknown> = { messages };
  if (options.conversation_id !== undefined) {
    payload.conversation_id = options.conversation_id;
  }
  if (options.from_slug !== undefined) {
    payload.from_slug = options.from_slug;
  }

  let response: Response;
  try {
    response = await fetch('/api/chat', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(payload),
      signal: options.signal,
    });
  } catch (err) {
    if ((err as { name?: string }).name === 'AbortError') return { ok: false };
    return { ok: false, errorCode: 'network' };
  }
  // ... rest unchanged from current implementation
}
```

The streaming-decode tail of `postChat` is unchanged.

## Test Strategy

**TDD discipline**: write tests first or alongside each implementation step. Schema test extensions before schema edit. `id.test.ts` before `id.ts` (trivial but the discipline is the contract). Validate / client / component / route tests interleave with the source edits.

**Expected new spec count**: ~22 specs across 5 test files (5 schema + 3 id + 4 validate + 3 client + 4 component + 6 route — actually 25; conservative 22 estimate).

**Coverage target**: ≥80% lines/functions/branches on the new module `src/lib/conversation/`. Trivially met (one function, three test paths exercise the named-export contract + uniqueness + regex). Coverage of modified files (schema, validate, client, component, route) does not regress — additions are exercised by the new specs.

**Edge cases covered**:
- Empty conversation (no submit) → no `newConversationId` call (component test).
- Multiple turns with the same id → threaded identically (route test).
- Malformed UUID at the wire boundary → 400 rejection (validate + route tests).
- `from_slug: null` (explicit) vs omitted (legacy) → both tolerated; `null` round-trips (schema + validate + client tests).
- Server-side fallback → real UUID v4 emitted (route regex assertion).
- Easter-egg path also gets the fields (route test) — ensures the new threading doesn't bypass the bypass.

**Not tested**:
- `crypto.randomUUID` collision probability — mathematically negligible (2^122 random bits per id; collision pigeonhole at billions of ids; Vercel's traffic ceiling makes this nonsensical to test for).
- Cross-tab id sharing — there is none. Each tab's `<ChatInterface>` mounts independently, mints its own id. By design.
- Concurrent submits with the same id — `useCallback` plus the `disabled={status === 'streaming'}` guard on the input prevents this at the UI layer; the server has no concurrency guard but logging is idempotent at the entry level (each entry is a separate JSONL line, race tradeoff already accepted in argue-log v1).

**Manual smoke** (post-merge, optional):
- Hit `/argue` on Vercel preview, send a message, check Vercel logs for an `argue-log/<today>.jsonl` write that contains a `conversation_id` field. Not gated by the dev-merge — the deploy proves the wire shape.
