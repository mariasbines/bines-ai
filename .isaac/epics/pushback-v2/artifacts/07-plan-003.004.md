# Implementation Plan: 003.004 — `/api/argue-judge/run` route

## Overview

The visitor-triggered judge endpoint. Edge POST route at `/api/argue-judge/run` with a novel auth model (IP-bind + 30-min recency + idempotency — no shared secret). Three risks mapped (PB2-SEC-002 IP-bind bypass, PB2-SEC-007 beacon tampering, PB2-OPS-003 race tolerance). Two new files: route + tests.

🔴 Full TDD. Medium complexity (single route, dense control flow, ten-case test matrix). 11 ACs. Single commit on `dev`.

After this story merges:
- POST to `/api/argue-judge/run` with `{ conversation_id }` triggers a Sonnet judge call when the IP matches the conversation's latest argue-log entry within 30 minutes AND no verdict yet exists.
- Idempotent replay returns the existing verdict.
- Errors return distinct HTTP codes (400, 403, 404, 410, 500, 502) — no 401 (no shared-secret model).
- No surface change for visitors (the chat-end beacon is story 003.005).
- No verdicts written until 003.005 fires the beacon — but the route is reachable by direct POST for testing/verification.

## Design decisions

### 1 · Auth model — IP-bind, recency, idempotency, NO shared secret

Locked architecture decision (`03-architecture.md:26`). The route is publicly callable from the visitor's own browser via `sendBeacon`. There's no shared secret — adding one would defeat the use case (visitors can't carry secrets to the chat-end). Instead the auth tuple is:

1. **IP-bind**: hashed request IP must match the latest `argue-log` entry's `ip_hash` for the supplied `conversation_id`. Salt-rotation tolerance via the try-current-then-previous pattern.
2. **Recency**: latest entry must be within 30 minutes of `now`.
3. **Idempotency**: an existing verdict for the `conversation_id` short-circuits the SDK call.

Distinct error codes prevent attacker oracle on which conversation IDs are real:
- 400 `validation` (malformed body)
- 403 `forbidden` (we know the id, your IP doesn't match)
- 404 `not_found` (we don't know the id, OR we know it but you're after the recency window — argue-log doesn't preserve old entries past 90 days, so 404 covers both)
- 410 `expired` (we know the id, it's within 90 days but past the 30-min judge window)
- 500 `upstream` (salt unset)
- 502 `upstream` (judge runner threw)

Wait — I'm conflating 404 and 410 in that prose. Re-reading AC-003 vs AC-004: 404 is "no entries match the conversation_id at all". 410 is "entries match but the latest is > 30 min old". Distinct. The spec is internally consistent.

### 2 · Edge runtime — `runtime = 'edge'`

Per AC-001 + scope rule "Do NOT use Node `crypto` or `fs`". Confirms the route imports from:
- `@/lib/argue-judge/storage` (uses `@vercel/blob` — Edge-compatible).
- `@/lib/argue-judge/runner` (uses `@anthropic-ai/sdk` — Edge-compatible).
- `@/lib/argue-log/storage` (Edge-compatible).
- `@/lib/argue-log/hash` (Web Crypto — Edge-compatible).
- `@/lib/argue-log/day` (pure — Edge-compatible).
- `next/server` (`NextRequest`, `NextResponse`).

Critical contrast with `/api/chat`: the chat route now uses Node runtime (story 003.002 — needed for `getFieldworkBySlug`). This route does NOT need Node runtime — it never reads MDX. Edge gives faster cold-start which matters more for a beacon endpoint (visitor tab might already be unloading).

### 3 · IP extraction — same pattern as existing routes

```ts
function getIp(req: Request): string {
  const xf = req.headers.get('x-forwarded-for');
  if (xf) return xf.split(',')[0].trim();
  return 'unknown';
}
```

Same as `src/app/api/chat/route.ts:60-62`. Consistency across routes; no novel surface.

### 4 · Salt-rotation tolerance

```ts
const saltCurrent = process.env.ARGUE_LOG_IP_SALT_CURRENT;
if (!saltCurrent) {
  console.error('[argue-judge] salt-unconfigured');
  return jsonResponse(500, { category: 'upstream' });
}
const saltPrevious = process.env.ARGUE_LOG_IP_SALT_PREVIOUS; // may be undefined

const ip = getIp(req);
const ipHashCurrent = await hashIp(ip, saltCurrent);
const ipHashPrevious = saltPrevious ? await hashIp(ip, saltPrevious) : null;

// IP-bind check (after recency check, see flow):
const matched =
  latest.ip_hash === ipHashCurrent ||
  (ipHashPrevious !== null && latest.ip_hash === ipHashPrevious);
if (!matched) return jsonResponse(403, { category: 'forbidden' });
```

The `ipHashPrevious === null` guard prevents an erroneous match against the empty string — if previous-salt isn't set, only current-salt comparison runs.

### 5 · Conversation lookup — today + yesterday only

```ts
const now = new Date();
const today = dayKeyUtc(now);
const yesterday = dayKeyUtc(new Date(now.getTime() - 24 * 60 * 60 * 1000));
const allEntries = [
  ...(await readArgueLogDay(today)),
  ...(await readArgueLogDay(yesterday)),
];
const entries = allEntries.filter((e) => e.conversation_id === body.conversation_id);
if (entries.length === 0) return jsonResponse(404, { category: 'not_found' });
```

Why two days: the 30-min recency window straddles midnight UTC for visitors hitting the route in the first half-hour of a UTC day. Reading both today + yesterday ensures we catch the latest entry no matter the wall-clock alignment.

### 6 · Sort order + recency check

```ts
entries.sort((a, b) => b.timestamp.localeCompare(a.timestamp)); // newest first
const latest = entries[0];
const ageMs = now.getTime() - new Date(latest.timestamp).getTime();
if (ageMs > 30 * 60 * 1000) {
  return jsonResponse(410, { category: 'expired' });
}
```

`Date(latest.timestamp).getTime()` is safe — schema enforces ISO datetime via `z.iso.datetime()`. Unparseable would have failed at write time. Defensive: if `getTime()` returns NaN somehow, `ageMs > 30 * 60 * 1000` evaluates to false, which is the wrong fail-direction (treats unparseable as fresh). Tighten with explicit NaN check:

```ts
const latestMs = new Date(latest.timestamp).getTime();
if (Number.isNaN(latestMs) || (now.getTime() - latestMs) > 30 * 60 * 1000) {
  return jsonResponse(410, { category: 'expired' });
}
```

Treat unparseable as expired — fail-closed.

### 7 · Idempotency before runner

```ts
const existing = await findVerdictByConversationId(body.conversation_id, { now });
if (existing !== null) {
  return jsonResponse(200, { judged: true, verdict: existing });
}
```

Critical: idempotency check happens AFTER auth (IP-bind + recency) — so an attacker who knows a conversation_id but doesn't have the right IP can't probe for verdict existence. Order: validate → salt → lookup entries → recency → IP-bind → idempotency → runner.

The architecture line 26 places idempotency in the "auth tuple" but ordering-wise it must come AFTER IP-bind for the no-oracle property.

### 8 · Runner call + storage write

```ts
const turnsAsc = entries
  .slice() // don't mutate the desc-sorted entries; copy.
  .sort((a, b) => a.timestamp.localeCompare(b.timestamp));
const turns = turnsAsc.flatMap((e) => e.turns);
const fromSlug = latest.from_slug ?? null;

let verdict: ArgueJudgeVerdict;
try {
  verdict = await judgeConversation(body.conversation_id, fromSlug, turns, { now });
} catch (err) {
  console.error(
    '[argue-judge] run-failed:',
    err instanceof Error ? err.name : 'unknown',
  );
  return jsonResponse(502, { category: 'upstream' });
}

await appendArgueJudge(verdict, { now });
return jsonResponse(200, { judged: true, verdict });
```

Notes:
- Turns are flattened across all argue-log entries for the conversation. Each entry contains `[user, assistant]` (or just `[user]` if refused). The judge needs the full conversation in chronological order.
- `latest.from_slug` is the canonical from_slug for the conversation. The architecture line 659 locks this: "the first entry's `from_slug` is canonical (concept Q3 — sticky at start)". Wait — first or latest? Sticky-at-start means first. But the spec on line 24 says "captured at chat start, sticky" — which means every entry's from_slug is the SAME, threaded from `<ChatInterface>`'s sticky useState. So latest === first in practice. Using `latest.from_slug` is fine because all entries share the same value.
- `appendArgueJudge` is awaited (not in `after()`). This route is short-lived, no streaming; the response shape includes the verdict, so the write must complete before the response.
- `now` is passed to both the runner (for `judged_at`) and storage (for the day-key) so they match.

### 9 · Response shape

Mirrors the chat route's pattern. AC-008 requires `X-Governed-By: bines.ai` on every response — all paths through `jsonResponse`.

## Test Strategy

**Spec count**: ~14-16 across one new test file.

**Coverage target**: ≥85% on `src/app/api/argue-judge/run/route.ts`. Each branch (validate / salt / lookup / recency / IP-bind / idempotency / runner / success) has at least one happy-path AND one failure-path spec; trivially exceeded.

**Edge cases covered**:
- Body that fails to JSON-parse (400 distinct from missing-field).
- Latest entry exactly at the 30-min boundary (within = ok; over = 410).
- Salt-previous unset entirely (only-current path).
- Salt-previous set but entry written under salt-current (current path matches first; previous never queried — code OR-shortcuts).
- Multiple argue-log entries spread across today + yesterday (recency uses the latest, not the earliest).
- `judgeConversation` throws → no `appendArgueJudge` call (transactional integrity).

**Not tested**:
- Real Sonnet API.
- Real Blob writes.
- Real Web Crypto (mocked via `hashIp` mock).
- End-to-end with a live `<ChatInterface>` posting via `sendBeacon` — verifiable post-merge on dev preview but not in unit-test scope.

## Risk mitigations (from `04-assessment.md`)

- **PB2-SEC-002** IP-bind bypass → AC-005 + Step 1 (try-current-then-previous match required); AC-002 + Design §4 (salt-rotation tolerance); AC-004 (30-min recency clamps the attack window); AC-006 (idempotency makes replay harmless); AC-009 + Design §1 (distinct error codes prevent oracle on which ids exist); AC-010 + Step 2 (full failure-mode test matrix including the salt-rotation-success regression spec).
- **PB2-SEC-007** beacon payload tampering → covered by PB2-SEC-002's IP-bind. A spoofed `conversation_id` from a different IP returns 403. A correct `conversation_id` from the original IP within 30 min IS the intended path; that's not tampering, that's the design.
- **PB2-OPS-003** race between idle beacon, pagehide beacon, and sweep → AC-006 + Design §7 (idempotency check returns 200 no-op on second fire — `findVerdictByConversationId` is the dedupe gate; matches the loader's dedupe-by-conversation_id at the read side).

## Clarifying questions

None unresolved. Three items deliberately scoped:

1. **Per-conversation lock to defeat double-fire even before idempotency** — out of scope. The architecture's "race tolerance via dedupe" is sufficient; lock-then-write would slow the hot path.
2. **`X-Robots-Tag: noindex` on the response** — not in story; the route is POST-only and beacon-fired, no crawler exposure. Skip.
3. **Per-IP rate limit** — not in story. The chat route's rate limit already throttles upstream argue-log writes; without an argue-log entry within 30 min there's nothing to judge, so the rate budget is implicitly enforced.
