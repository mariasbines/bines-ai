# Implementation Plan: 003.006 — `/api/argue-judge/sweep` cron + `vercel.json`

## Overview

The resilience net for the judge. Daily Vercel cron at `/api/argue-judge/sweep` (GET, edge, `CRON_SECRET` bearer auth) reads yesterday's argue-log entries, skips conversations already judged, runs `judgeConversation` for the rest, writes verdicts. Per-conversation isolation — one runner failure doesn't kill the sweep. Schedule `30 4 * * *` UTC (one hour after argue-log/cleanup).

**Last Phase B story.** After this merges, the 7-day soft-launch gap begins.

🟡 Critical Path TDD. Medium complexity. 8 ACs. Four risks (PB2-SEC-005, PB2-OPS-001, PB2-OPS-002, PB2-PRF-002). Single commit on `dev`.

## Design decisions

### 1 · Auth — mirror cleanup, accept secret-reuse risk

Per AC-001 and the architecture's PB2-SEC-005 mitigation: reuses the existing `CRON_SECRET` rather than provisioning a separate one. Operational simplicity outweighs separate-secrets benefit at v1 scale.

`timingSafeEqual` helper is duplicated inline (same 8-line implementation as cleanup route — one of these days extract to a shared `src/lib/timing-safe.ts`, but architecture says "small helper inline" so do that). Same ASCII-only-secret discipline (CRON_SECRET is hex-encoded; `charCodeAt` is exact for BMP-range).

```ts
function timingSafeEqual(a: string, b: string): boolean {
  if (a.length !== b.length) return false;
  let diff = 0;
  for (let i = 0; i < a.length; i++) {
    diff |= a.charCodeAt(i) ^ b.charCodeAt(i);
  }
  return diff === 0;
}
```

Auth flow:
- 500 if `CRON_SECRET` env unset (`[argue-judge] cron-secret-unconfigured`).
- 401 if `Authorization` header missing, malformed (no `Bearer ` prefix), or token doesn't match.

### 2 · Target-day resolution — yesterday default + `?day=` override

```ts
const url = new URL(req.url);
const queryDay = url.searchParams.get('day');
let targetDay: string;
if (queryDay !== null) {
  if (!isValidDayKey(queryDay)) {
    return jsonResponse(400, { category: 'validation' });
  }
  targetDay = queryDay;
} else {
  const yesterday = new Date(Date.now() - 24 * 60 * 60 * 1000);
  targetDay = dayKeyUtc(yesterday);
}
```

The `?day=` override exists for manual replay. Maria's runbook (in `docs/argue-judge-ops.md`) documents the curl command.

### 3 · Selection logic — skip already-judged + skip pre-Phase-A entries

```ts
const entries = await readArgueLogDay(targetDay);

// Group by conversation_id, skipping pre-Phase-A entries.
const groups = new Map<string, typeof entries>();
for (const e of entries) {
  if (!e.conversation_id) continue; // pre-Phase-A; ages out via cleanup
  const arr = groups.get(e.conversation_id) ?? [];
  arr.push(e);
  groups.set(e.conversation_id, arr);
}

// Existing-judgments lookup spans target day + today (verdict day-key is
// when the JUDGE ran, not when the conversation happened — verdicts may
// have landed on either day).
const today = dayKeyUtc();
const existingDays = targetDay === today ? [targetDay] : [targetDay, today];
const existingJudges = (
  await Promise.all(existingDays.map((d) => readArgueJudgeDay(d)))
).flat();
const judgedIds = new Set(existingJudges.map((v) => v.conversation_id));

// Iterate groups; skip judged; run + write the rest with per-conversation
// isolation.
let judged = 0;
let skippedAlreadyJudged = 0;
let errors = 0;
for (const [convId, convEntries] of groups) {
  if (judgedIds.has(convId)) {
    skippedAlreadyJudged++;
    continue;
  }
  try {
    convEntries.sort((a, b) => a.timestamp.localeCompare(b.timestamp));
    const turns = convEntries.flatMap((e) => e.turns);
    const fromSlug = convEntries[0]?.from_slug ?? null;
    const verdict = await judgeConversation(convId, fromSlug, turns);
    await appendArgueJudge(verdict);
    judged++;
  } catch (err) {
    errors++;
    console.error(
      `[argue-judge] sweep-failed conv_id=${convId} reason=${err instanceof Error ? err.name : 'unknown'}`,
    );
    // Continue — per-conversation isolation. A flaky Sonnet API blip
    // cannot kill the whole nightly job.
  }
}
```

Notes:
- `convEntries[0]` is the earliest entry after asc-sort — the canonical from_slug per architecture line 659 ("the first entry's `from_slug` is canonical (concept Q3 — sticky at start)"). Note: differs from the run route which uses `latest.from_slug`. Both are correct because the from_slug is sticky for the conversation; using the first entry here is the architecturally documented pattern. (In practice all entries in a conversation share the same value.)
- Edge case: `convEntries[0]?.from_slug ?? null` — defensive `?.` even though group construction guarantees ≥1 entry per group.
- `judgedIds` is built once before iteration — race with concurrent run-route writes is accepted (the loader-side dedupe handles double verdicts; sweep just skips on next run).

### 4 · Response envelope

```ts
return jsonResponse(200, {
  day: targetDay,
  judged,
  skipped_already_judged: skippedAlreadyJudged,
  errors,
});
```

`X-Governed-By: bines.ai` on every response (consistent with cleanup + run).

### 5 · `vercel.json` modification

Per AC-006:

```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "crons": [
    { "path": "/api/argue-log/cleanup", "schedule": "30 3 * * *" },
    { "path": "/api/argue-judge/sweep", "schedule": "30 4 * * *" }
  ],
  "functions": {
    "src/app/api/argue-judge/sweep/route.ts": {
      "maxDuration": 300
    }
  }
}
```

`maxDuration: 300` (5 min) covers up to ~150 conversations/day at sequential pacing (~1.5-2s per `judgeConversation` call). Architecture documents the threshold for moving to concurrency at ~30/day (we have lots of headroom).

The existing `argue-log/cleanup` cron entry is preserved verbatim.

### 6 · `docs/argue-judge-ops.md`

Per AC-007. Five sections per the existing `docs/argue-log-ops.md` pattern:

1. **Cron schedule + ordering rationale.** `30 4 * * *` UTC — one hour after `argue-log/cleanup`. Ordering matters: cleanup deletes argue-log entries older than 90 days BEFORE sweep runs, so sweep never sees a soon-to-be-deleted entry. (At v1 scale, sweep targets yesterday — the 90-day-cleanup boundary is far away — but the ordering is good hygiene.)

2. **Manual replay command.**
   ```bash
   curl -H "Authorization: Bearer $CRON_SECRET" "https://bines.ai/api/argue-judge/sweep?day=YYYY-MM-DD"
   ```

3. **Monthly cron-history check.** Maria spot-checks the Vercel dashboard's cron tab; any "missed run" surfaces there.

4. **GDPR erasure path for verdicts.** `argue-judges/<day>.jsonl` blobs are deletable via the same Blob admin pattern as `argue-log/<day>.jsonl`. After deletion, the next deploy rebuilds without those excerpts. Document the manual-delete command (uses the existing argue-log cleanup path as a model — sweep deletion of argue-judges is a manual-only path; no automated cron deletes verdicts).

5. **Cost note.** At ~20 conversations/day, nightly Sonnet cost ~$0.20. If conversations/day grows past ~50, watch the Anthropic billing dashboard and consider concurrency limits + per-IP throttling on the run route.

Plus a brief mention of the **fail-shut policy**: sweep errors do not retry within the same run; the next night's sweep picks up un-judged conversations. Verdicts age out only via manual day-blob deletion (no automated retention cron yet).

British English; lowercase headers consistent with `docs/argue-log-ops.md`.

## Test Strategy

**Spec count**: ~13-15 across the new test file.

**Coverage**: ≥80% on `route.ts`. Each branch — auth, day-override, selection, error-isolation — has happy + sad paths.

**Edge cases**:
- `?day=today` (technically valid; sweep targeting today is unusual but legal).
- Sequential `judgeConversation` failures (multiple errors counted).
- Mixed groups (some with valid conversation_id, some pre-Phase-A) → only valid ones counted.
- Argue-log day with one conversation spanning multiple entries (sweep groups them).

**Not tested**:
- Real Vercel cron invocation — manual smoke test post-merge (verify the schedule appears in dashboard).
- Real Sonnet API — mocked.
- Real Blob writes — mocked.
- `maxDuration: 300` actually applied at deploy — verifiable post-merge but not in unit tests.

- **PB2-PRF-002** sweep cron Sonnet-call serialisation → AC-006 + Step 3 (`maxDuration: 300` covers up to ~150 conversations/day at sequential pacing).

## Rollback Plan

- `git revert` the merge commit on `dev`.
- New route file disappears.
- `vercel.json` reverts to the single-cron shape — cleanup cron continues to fire; sweep cron disappears (Vercel removes it on next deploy).
- `docs/argue-judge-ops.md` disappears.
- Already-judged verdicts are unaffected (no deletions occur during revert).
- 003.007+ (Phase C) hasn't shipped to dev yet; no downstream breakage.

Zero data loss. Zero downtime. The only consequence of revert is that sweeps stop running — un-judged conversations from the chat-end signal (003.005) accumulate until the route is restored.
